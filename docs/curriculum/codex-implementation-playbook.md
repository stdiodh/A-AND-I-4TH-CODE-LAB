# Codex Implementation Playbook

## 문서 목적

이 문서는 Codex가 A&I 백엔드 실습 레포를 만들거나 보정할 때 따라야 하는 실행 순서를 고정합니다.

핵심은 아래 세 가지입니다.

1. Codex가 항상 같은 순서로 중앙 문서를 읽게 한다.
2. 구현 범위를 과하게 넓히지 않게 한다.
3. 이론보다 구현 순서와 체크리스트를 먼저 보게 한다.

---

## Codex 문서 읽기 순서

Codex는 항상 아래 순서로 문서를 읽습니다.

1. 루트 `README.md`
2. `docs/curriculum/a-and-i-backend-curriculum-master-plan.md`
3. `docs/curriculum/a-and-i-backend-implementation-scope-plan.md`
4. `docs/curriculum/codex-implementation-playbook.md`
5. `docs/curriculum/sequence-execution-protocol.md`
6. `docs/guides/implementation-writing-guide.md`
7. `docs/guides/theory-writing-guide.md`
8. `docs/guides/theory-doc-template.md`
9. `docs/guides/checklist-writing-guide.md`
10. `docs/sequences/0X-...md`
11. 그 다음 해당 토픽 레포의 `README.md`, `docs/theory.md`, `docs/implementation.md`, `docs/checklist.md`, starter 코드

즉, Codex는 아래 순서로만 움직입니다.

1. 현재 작업 상태와 현재 시퀀스 확인
2. 전체 방향 확인
3. 구현 범위 원칙 확인
4. 문서 작성 원칙 확인
5. 해당 도메인 문서 확인
6. 실제 레포 구조와 starter 코드 확인
7. TODO 설계 및 문서/코드 생성

Codex는 문서 종류에 따라 아래 가이드를 반드시 따라야 합니다.

"어떤 문서를 쓰는가"에 따라 먼저 참조 문서를 확인해야 합니다.

## Codex가 반드시 지켜야 할 기준

### 1. 이론 문서를 중앙 문서에 다시 쓰지 않습니다

상세 이론은 각 토픽 레포 내부 문서에서 다룹니다.

중앙 문서에는 아래만 둡니다.

- 구현 범위
- 구현 순서
- TODO 기준
- 체크리스트
- 다음 도메인 연결 포인트

### 1-1. 현재 시퀀스 외 작업은 금지합니다

Codex는 항상 현재 시퀀스 하나만 작업합니다.

- 다음 시퀀스를 미리 만들지 않습니다.
- 사용자 승인 없이 다음 시퀀스로 넘어가지 않습니다.
- 현재 시퀀스와 무관한 파일을 함께 생성하지 않습니다.

### 1-2. 주제가 분리되면 레포도 분리합니다

Codex는 학습 주제가 분명히 갈라질 때,
하나의 레포 안에서 선택형으로 억지로 묶지 않습니다.

예:

- OAuth2 로그인
- Email Verification

이처럼 별도 트랙으로 운영하는 편이 더 자연스러운 주제는
별도 레포로 분리하는 것을 기본 원칙으로 봅니다.

### 1-3. 시퀀스 브랜치는 `NN-implementation`, `NN-answer`로 운영합니다

Codex는 시퀀스 작업 시 아래 브랜치 구조를 기본으로 사용합니다.

- `NN-implementation`
- `NN-answer`

예:

- `02-implementation`, `02-answer`
- `03-implementation`, `03-answer`

즉, 단일 `implementation`, `answer` 브랜치를 계속 덮어쓰는 방식으로 운영하지 않습니다.

### 1-4. 각 서브모듈의 `main` 브랜치는 안내 브랜치입니다

Codex는 각 서브모듈 레포의 `main` 브랜치를
실습 starter 브랜치가 아니라 안내 브랜치로 유지해야 합니다.

`main`에는 최소한 아래가 있어야 합니다.

- 레포 소개
- 문서 목록
- 브랜치 안내
- 학생 시작 브랜치 / 강사 비교 브랜치 안내

즉, 사용자가 `main`에 들어오면
"이 레포는 무엇이고 어디로 가야 하는가"를 바로 이해할 수 있어야 합니다.

### 2. 학생 구현 범위는 핵심 흐름만 남깁니다

Codex는 아래를 학생 구현 대상으로 우선 선택합니다.

- 요청/응답 흐름의 핵심 코드
- 다음 단계에서도 다시 참조할 구조
- 수업 시간 안에 완성 가능한 핵심 코드
- 실행 결과가 바로 확인 가능한 코드

Codex는 아래를 가능하면 미리 제공합니다.

- 반복 설정
- 환경 의존성이 큰 인프라 설정
- 긴 보일러플레이트
- 과도한 예외 케이스
- 복잡한 운영 스크립트 전체

### 3. TODO는 순서형 힌트로 작성합니다

좋은 TODO는 아래 조건을 만족합니다.

- 한 단계씩 순서가 보인다
- 학생이 다음 행동을 바로 알 수 있다
- 결과 확인 포인트가 포함된다

좋지 않은 TODO 예시:

```kotlin
// TODO: JWT 구현
```

좋은 TODO 예시:

```kotlin
// TODO 1. 로그인 요청에서 email 값을 꺼내세요.
// TODO 2. 저장소에서 사용자를 조회하세요.
// TODO 3. 비밀번호를 검증하세요.
// TODO 4. JWT를 발급해 LoginResponse로 반환하세요.
```

### 3-1. 생성 전에 산출물 목록을 먼저 요약합니다

Codex는 파일을 만들기 전에 아래를 먼저 짧게 정리합니다.

- 현재 시퀀스
- 이번 작업 범위
- 생성할 파일 목록
- 이번 턴에서 만들지 않을 범위

### 3-2. 큰 작업은 계획서를 먼저 보여줍니다

Codex는 아래 조건 중 하나라도 해당하면
바로 구현하지 않고 먼저 계획서를 사용자에게 보여줘야 합니다.

- 브랜치나 레포 구조를 다시 짜야 할 때
- 여러 문서를 함께 고쳐야 할 때
- 문서와 코드가 동시에 흔들릴 때
- 시퀀스 전체 구조를 다시 잡아야 할 때

계획서에는 최소한 아래가 들어가야 합니다.

1. 현재 문제 진단
2. 수정 범위
3. 작업 순서
4. 이번 턴에서 바꾸는 것 / 바꾸지 않는 것

### 3-3. 이해가 불충분하면 질문을 먼저 합니다

Codex는 사용자 요구를 정확히 이해하지 못했다고 판단되면
추측으로 진행하지 않고 질문을 먼저 해야 합니다.

질문이 필요한 대표 상황:

- 어떤 시퀀스를 고정 기준으로 삼을지 모를 때
- 레포를 분리할지 기존 레포를 재사용할지 애매할 때
- 문서 목적이 학생용인지 강사용인지 불분명할 때
- 현재 턴에서 코드까지 바꿀지 문서만 바꿀지 불분명할 때

### 4. 문서는 주차명이 아니라 도메인 이름으로 씁니다

중앙 문서와 구현 설계 문서는 `Week 13` 같은 제목을 쓰지 않습니다.

아래처럼 도메인이 바로 보이게 씁니다.

- 요청-응답과 메모리 CRUD
- 영속성 저장과 계층 분리
- 안전한 요청 처리
- 인증과 JWT

### 5. 각 도메인 문서에는 두 종류의 체크리스트가 들어갑니다

- 학생 체크리스트
- 강사 / PPT 체크리스트

Codex는 이 둘 중 하나만 작성하면 안 됩니다.

### 6. 구현 문서와 이론 문서는 서로 다른 기준으로 씁니다

- 구현 문서는 결과보다 순서를 먼저 보여줘야 합니다.
- 이론 문서는 정의보다 문제 상황과 코드 연결을 먼저 보여줘야 합니다.
- 이론 문서는 반드시 템플릿 구조를 확인한 뒤 작성합니다.
- 두 문서의 문장 구조와 목적을 섞지 않습니다.

### 6-1. 실무 확장 개념은 자동으로 같이 문서화합니다

`02`부터는 각 시퀀스에 실무 확장 개념 1개가 붙습니다.
이 규칙은 이후 시퀀스에서도 별도 지시 없이 자동으로 적용합니다.

Codex는 실무 확장 개념이 있는 시퀀스를 작업할 때 아래를 함께 갱신해야 합니다.

- 루트 `docs/sequences/0X-...md`
- 토픽 레포 `docs/theory.md`
- 토픽 레포 `docs/implementation.md`
- 토픽 레포 `docs/answer-guide.md`
- 필요 시 `docs/checklist.md`

그리고 이론 문서에는 아래가 모두 있어야 합니다.

1. 문제 상황
2. 문제 코드
3. 내부 동작 또는 예상 쿼리 흐름
4. 왜 비용이 커지는지
5. 대표 해결 코드 예시
6. 이번 시퀀스의 구현 범위

구현 문서와 정답 가이드는
이 실무 확장 개념이 이번 단계의 어떤 코드와 연결되는지까지 이어줘야 합니다.

즉, `theory.md`에만 깊게 쓰고 끝내지 않고,
학생이 어느 문서를 보든 흐름이 이어지게 만들어야 합니다.

### 6-2. 실무 확장 개념이 기초 설명을 밀어내면 안 됩니다

Codex는 실무 확장 개념을 강화할 때,
아래 실수를 하면 안 됩니다.

- 기초 개념 설명 삭제
- 현재 코드 흐름 설명 삭제
- 용어 설명 삭제
- "실무 팁"만 남기고 학생 입문 설명 제거

모든 `theory.md`는 아래 3층 구조를 유지해야 합니다.

1. 기초 이론
2. 현재 코드 흐름
3. 실무 확장 개념

즉, 실무 확장 개념은 추가 레이어이지,
기초 설명을 대체하는 레이어가 아닙니다.

Codex는 문서 수정 후 아래 질문을 스스로 확인해야 합니다.

1. 이 문서만 읽어도 학생이 기본 개념을 처음 이해할 수 있는가
2. 현재 코드에서 어디를 보면 되는지 설명하는가
3. 실무 확장 개념이 그 위에 자연스럽게 얹혀 있는가

이 질문 중 하나라도 `아니오`면,
문서는 다시 써야 합니다.

### 7. 브랜치가 바뀌면 docs도 반드시 바뀌어야 합니다

Codex는 아래를 반드시 지켜야 합니다.

- `02` 브랜치의 `docs/theory.md`와 `03` 브랜치의 `docs/theory.md`는 같으면 안 됩니다.
- `NN-implementation`과 `NN-answer`는 같은 시퀀스를 설명해야 합니다.
- 이전 시퀀스 문서를 복사해두고 제목만 바꾸는 식으로 끝내면 안 됩니다.

즉, 시퀀스 브랜치가 바뀌면
`README.md`, `docs/theory.md`, `docs/implementation.md`, `docs/answer-guide.md`, `docs/checklist.md`, `docs/assets.md`도 함께 현재 시퀀스에 맞게 교체되어야 합니다.

### 8. `main` 브랜치도 문서 유지 대상입니다

Codex는 시퀀스 브랜치만 보지 않고,
서브모듈 `main` 브랜치의 안내 문서도 함께 유지해야 합니다.

권장 구성:

- `README.md`
- `docs/branch-guide.md`
- `docs/sequence-map.md`

이 문서는 시퀀스 문서와 다른 역할을 하며,
레포 전체를 요약하고 브랜치 진입 경로를 설명해야 합니다.

## Codex의 출력 기준

Codex가 실습 레포용 문서를 만들 때는 아래 순서로 구성합니다.

1. 이 도메인이 필요한 이유
2. 학생이 완성할 최종 흐름
3. 학생이 직접 구현할 순서
4. TODO를 넣을 파일
5. 미리 제공할 것
6. 학생 체크리스트
7. 강사 / PPT 체크리스트
8. 다음 도메인 연결 포인트

그리고 한 시퀀스 작업이 끝나면 아래 순서로 마무리합니다.

1. README 링크와 상태를 갱신한다.
2. 완료 조건 충족 여부를 확인한다.
3. 다음 시퀀스로 가지 않고 사용자 승인을 기다린다.

---

## 최종 기준

Codex는 "무엇을 구현할까"보다
"학생이 어떤 순서로 구현하면 흐름을 이해할 수 있을까"를 먼저 판단해야 합니다.
