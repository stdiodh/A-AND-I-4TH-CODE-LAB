# A&I 백엔드 커리큘럼 구현 범위 계획서

## 문서 목적

이 문서는 A&I 백엔드 커리큘럼을 입문자 기준으로 운영할 때,
각 도메인별 구현 범위를 어떻게 잡을지 정리한 기준 문서입니다.

이 문서의 핵심 목표는 아래와 같습니다.

1. Codex가 실습 레포를 만들 때 구현 범위를 과하거나 부족하지 않게 잡을 수 있어야 한다.
2. 학생이 어떤 순서로 코드를 작성해야 하는지 한눈에 보이게 해야 한다.
3. 반복 설정과 무거운 부가 기능은 제공하고, 핵심 흐름만 직접 구현하게 해야 한다.
4. 강사가 수업이나 PPT를 준비할 때 이 문서의 체크리스트만으로 빠짐없이 준비할 수 있어야 한다.

이 문서는 이론 문서를 대신하지 않습니다.
각 토픽 레포의 이론은 해당 레포 내부 문서에서 다루고,
중앙 문서는 구현 범위와 진행 순서를 고정하는 역할만 맡습니다.

---

## 이 문서의 사용 방식

이 문서는 주차명 기준이 아니라, 학습 도메인 순서 기준으로 사용합니다.

즉, `week13`, `week14`처럼 보지 않고 아래처럼 봅니다.

- 요청/응답과 메모리 CRUD
- 영속성 저장과 계층 분리
- 안전한 요청 처리
- 인증과 JWT
- 테스트
- 캐시
- 실시간 통신
- 배포와 운영
- 자동화
- 리팩토링
- 이벤트 기반 흐름

Codex는 실습 레포를 만들기 전에 아래 순서로 문서를 참고합니다.

1. 루트 `README.md`
2. 커리큘럼 전체 운영안
3. 구현 범위 계획서
4. Codex playbook
5. 시퀀스 실행 프로토콜
6. 구현 문서 작성 가이드
7. 이론 문서 템플릿
8. 이론 문서 작성 가이드
9. 체크리스트 작성 가이드
10. 해당 도메인 순서 문서
11. 해당 레포 이론 문서 및 starter 코드

## 핵심 운영 원칙

### 1. 학생은 핵심 흐름만 직접 구현합니다

학생이 직접 구현하는 코드는 아래 기준으로 고릅니다.

- 그 도메인의 핵심 개념을 가장 잘 드러내는가
- 다음 단계에서도 다시 떠올려야 하는가
- 수업 시간 안에 끝낼 수 있는가
- 실행 결과를 바로 확인할 수 있는가

### 2. 아래 항목은 가급적 미리 제공합니다

- 반복적인 설정 코드
- 환경 의존성이 큰 인프라 설정
- 학습 목표와 직접 관련 없는 부가 기능
- 입문자에게 과도한 예외 상황 처리
- 긴 보일러플레이트 코드
- 복잡한 배포 스크립트 전체

### 3. 구현 문서는 결과가 아니라 순서를 보여줘야 합니다

좋지 않은 방식:

- Service를 만든다
- Controller를 만든다

좋은 방식:

1. 요청 DTO를 만든다.
2. 저장할 데이터 클래스를 만든다.
3. 저장소를 만든다.
4. Service에서 저장소를 호출한다.
5. Controller에서 Service를 연결한다.
6. Swagger 또는 테스트로 결과를 확인한다.

### 4. TODO는 순서형 힌트로 작성합니다

좋지 않은 TODO:

```kotlin
// TODO: 서비스 로직 구현
```

좋은 TODO:

```kotlin
// TODO 1. request에서 title, content 값을 꺼내세요.
// TODO 2. 이 값으로 Post 객체를 만드세요.
// TODO 3. repository.save(...)를 호출하세요.
// TODO 4. 저장 결과를 PostResponse로 변환하세요.
```

### 5. 각 도메인 문서에는 체크리스트가 반드시 들어갑니다

각 도메인 문서에는 아래 두 종류의 체크리스트가 있어야 합니다.

- 학생 체크리스트
- 강사 / 발표 / PPT 체크리스트

### 6. 한 번에 하나의 시퀀스만 작업합니다

구현 범위가 흔들리지 않도록 아래 원칙을 고정합니다.

- 현재 시퀀스 하나만 작업합니다.
- 다음 시퀀스는 사용자 승인 전 시작하지 않습니다.
- 생성 전에 이번 턴의 산출물 목록을 먼저 정리합니다.
- 작업 후에는 README 상태를 반드시 갱신합니다.

### 6-1. 시퀀스마다 브랜치를 따로 나눕니다

각 시퀀스는 아래 브랜치 쌍을 기본으로 사용합니다.

- `NN-implementation`
- `NN-answer`

예:

- `02-implementation`, `02-answer`
- `03-implementation`, `03-answer`

학생은 `NN-implementation`에서 작업하고,
강사는 `NN-answer`를 기준으로 비교합니다.

### 6-2. 브랜치별 docs는 시퀀스에 따라 계속 바뀌어야 합니다

각 브랜치에는 아래 문서가 있어야 합니다.

- `README.md`
- `docs/theory.md`
- `docs/implementation.md`
- `docs/answer-guide.md`
- `docs/checklist.md`
- `docs/assets.md`

그리고 이 문서는 반드시 현재 시퀀스 내용만 설명해야 합니다.

즉:

- `02` 이론 문서를 `03` 브랜치에 그대로 두면 안 됩니다.
- `03` 체크리스트가 `02` 설명을 계속 담고 있으면 안 됩니다.
- 문서는 시퀀스 순서에 따라 매번 교체되어야 합니다.

### 6-3. 서브모듈 `main` 브랜치에는 레포 안내 문서를 둡니다

각 서브모듈 레포의 `main` 브랜치에는
아래 안내 문서가 있어야 합니다.

- `README.md`
- 필요 시 `docs/branch-guide.md`
- 필요 시 `docs/sequence-map.md`

이 문서는 아래를 설명해야 합니다.

- 이 레포가 어떤 시퀀스를 다루는지
- 어떤 문서가 있는지
- 어떤 브랜치에서 실습을 시작해야 하는지
- 어떤 브랜치에서 정답을 비교해야 하는지

### 7. 큰 작업은 실행 전 계획서를 먼저 보여줍니다

아래 상황에서는 바로 구현하지 않고,
먼저 사용자에게 계획서를 보여줘야 합니다.

- 레포 구조를 나눠야 할 때
- 브랜치 전략을 바꿔야 할 때
- 여러 시퀀스 문서가 함께 흔들릴 때
- 문서와 코드 구조를 동시에 다시 잡아야 할 때

계획서에는 아래 항목이 들어가야 합니다.

1. 현재 문제
2. 수정 범위
3. 작업 순서
4. 이번 턴에 바꾸는 것과 다음 턴으로 넘기는 것

### 8. 사용자 의도를 이해하지 못하면 질문을 먼저 합니다

Codex는 아래 상황에서 추측으로 진행하지 않습니다.

- 어떤 시퀀스를 기준으로 삼아야 할지 애매할 때
- 문서를 먼저 고칠지 코드를 먼저 고칠지 애매할 때
- 별도 레포 분리가 필요한지 확신할 수 없을 때
- 학생용 문서와 강사용 문서의 경계가 흐릴 때

이 경우에는 무조건 사용자에게 질문하고,
확인된 뒤에만 작업을 진행합니다.

## 각 도메인 문서 템플릿

각 `docs/sequences/*.md` 문서는 아래 형식을 따릅니다.

1. 도메인 이름
2. 이 도메인이 필요한 이유
3. 학생이 완성할 최종 흐름
4. 학생이 직접 구현할 순서
5. TODO를 넣을 파일
6. 미리 제공할 것
7. 학생 체크리스트
8. 강사 / PPT 체크리스트
9. 다음 도메인 연결 포인트

그리고 각 문서는 아래 가이드를 반드시 따릅니다.

- `theory.md`: `docs/guides/theory-doc-template.md`, `docs/guides/theory-writing-guide.md`
- `implementation.md`: `docs/guides/implementation-writing-guide.md`
- `checklist.md`: `docs/guides/checklist-writing-guide.md`
- 브랜치/레포/완료 순서: `docs/curriculum/sequence-execution-protocol.md`
