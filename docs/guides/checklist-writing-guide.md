# 체크리스트 작성 가이드

## 문서 목적

이 문서는 각 도메인 문서에 들어갈 체크리스트를 어떤 기준으로 작성할지 정리합니다.

체크리스트는 부록이 아니라,
학생과 강사가 무엇을 확인해야 하는지 빠르게 잡아주는 핵심 도구입니다.

---

## 체크리스트의 종류

각 도메인 문서에는 아래 두 종류의 체크리스트가 반드시 있어야 합니다.

### 1. 학생 체크리스트

학생이 직접 실습 후 확인해야 할 항목입니다.

좋은 학생 체크리스트의 특징:

- 실행 결과를 확인할 수 있다
- 개념을 말로 설명할 수 있다
- 정상/실패 흐름을 둘 다 확인하게 만든다

### 2. 강사 / PPT 체크리스트

강사가 수업 자료나 발표 자료를 준비할 때 빠뜨리면 안 되는 항목입니다.

좋은 강사 체크리스트의 특징:

- 반드시 설명해야 할 개념이 있다
- 그림으로 보여줄 흐름이 있다
- 시연 포인트가 있다
- 학생이 헷갈릴 포인트를 미리 막아준다

---

## 학생 체크리스트 작성 규칙

좋은 예시:

- [ ] Swagger에서 POST와 GET을 직접 실행할 수 있다.
- [ ] 메모리 저장과 DB 저장의 차이를 설명할 수 있다.
- [ ] Validation 실패 응답을 직접 확인했다.

나쁜 예시:

- [ ] JPA를 이해했다.
- [ ] Spring Security를 배웠다.

학생 체크리스트는 추상적인 표현보다
직접 확인 가능한 질문 형태가 좋습니다.

---

## 강사 / PPT 체크리스트 작성 규칙

좋은 예시:

- [ ] 요청 → Controller → Service → 응답 흐름 그림이 있는가
- [ ] Validation 실패 사례를 준비했는가
- [ ] 토큰 유무에 따른 접근 차이를 시연할 수 있는가

나쁜 예시:

- [ ] 발표 준비
- [ ] 자료 확인

강사 체크리스트는 "무엇을 준비해야 하는가"가 바로 보여야 합니다.

---

## 체크리스트 개수 기준

권장 기준:

- 학생 체크리스트: 3~5개
- 강사 / PPT 체크리스트: 3~5개

너무 많으면 핵심이 흐려지고,
너무 적으면 실제 준비에 도움이 되지 않습니다.

---

## 체크리스트 작성 질문

학생 체크리스트를 쓸 때는 아래를 먼저 묻습니다.

- 학생이 직접 확인할 수 있는가
- 실행 결과로 검증 가능한가
- 말로 설명할 수 있는 개념인가

강사 체크리스트를 쓸 때는 아래를 먼저 묻습니다.

- 이 개념을 반드시 설명해야 하는가
- 그림이나 흐름으로 보여줘야 하는가
- 시연하지 않으면 추상적으로 느껴지는가

---

## 최종 기준

좋은 체크리스트는 문서를 읽은 뒤에 붙이는 부가 항목이 아니라,
학생과 강사가 무엇을 놓치지 말아야 하는지를 고정하는 운영 장치입니다.
