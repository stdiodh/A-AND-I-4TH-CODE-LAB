## 중요한 코드 먼저 보기

> 이 섹션은 필수입니다.  
> 긴 코드를 다 보여주기보다, **핵심 흐름이 보이는 짧은 코드**를 먼저 보여줍니다.  
> 그리고 코드 블록 안에는 학생이 바로 이해할 수 있도록 **짧은 주석**을 넣습니다.

### 1. [요청이 처음 들어오는 코드]

```kotlin
@RestController
@RequestMapping("/[resources]")
class [Resource]Controller(
    private val [resource]Service: [Resource]Service
) {

    @PostMapping
    fun create(
        @RequestBody request: [Resource]CreateRequest
        // 사용자가 보낸 요청 데이터를 먼저 받습니다.
    ): ResponseEntity<[Resource]Response> {

        val response = [resource]Service.create(request)
        // 실제 처리 로직은 Controller가 직접 하지 않고 Service에 맡깁니다.

        return ResponseEntity.ok(response)
        // 처리 결과를 다시 클라이언트에게 응답으로 돌려줍니다.
    }
}
```

- 이 코드는 **요청의 시작점**을 보여줍니다.
- 여기서는 특히 `@[Post/Get/Put/Delete]Mapping`과 `service.[method](...)`를 먼저 보세요.
- 학생이 기억해야 할 핵심은 **“Controller는 요청을 받고, 실제 처리는 Service에 넘긴다”**입니다.
- 파일: `[src/main/kotlin/.../[Resource]Controller.kt]`

---

### 2. [실제 처리 흐름이 모이는 코드]

```kotlin
@Service
class [Resource]Service(
    private val [resource]Repository: [Resource]Repository
) {

    fun create(request: [Resource]CreateRequest): [Resource]Response {
        val entity = [Resource](
            name = request.name,
            description = request.description
            // 요청 DTO에서 필요한 값만 꺼내 Entity를 만듭니다.
        )

        val saved = [resource]Repository.save(entity)
        // 이 줄에서 실제로 DB 저장이 일어납니다.

        return [Resource]Response(
            id = saved.id!!,
            name = saved.name,
            description = saved.description
            // 저장 결과를 그대로 주지 않고, 응답 DTO로 다시 감싸서 돌려줍니다.
        )
    }
}
```

- 이 코드는 **실제 비즈니스 처리 흐름**을 보여줍니다.
- 여기서는 특히 `repository.save(...)` 또는 핵심 처리 메서드를 먼저 보세요.
- 학생이 기억해야 할 핵심은 **“Service는 요청을 처리해서 저장/조회하고, 응답 형태까지 정리한다”**입니다.
- 파일: `[src/main/kotlin/.../[Resource]Service.kt]`

### 이번 시퀀스의 실무 확장 개념

- [예: 관계 매핑]
- [예: N+1 문제]

### 1. 문제 상황 먼저 잡기

[이 개념이 실무에서 언제 문제로 보이기 시작하는지 설명합니다.]

예시 문장:
- 처음에는 [단순한 흐름]만 있어도 충분해 보입니다.
- 그런데 [연관 데이터 / 추가 조건 / 운영 환경 / 규모 증가]가 들어오면 문제가 보이기 시작합니다.
- 이번 시퀀스에서는 그 시작점을 이해하는 데 집중합니다.

### 2. 처음 보면 자연스러운 코드

```kotlin
val [items] = [repository].[findMethod]()

return [items].map { item ->
    [Response](
        id = item.id,
        relatedCount = item.[relatedValues].size
        // 여기처럼 연관 데이터를 순회하거나 접근하는 코드가
        // 처음에는 자연스럽게 보여도 추가 쿼리를 만들 수 있습니다.
    )
}
```

- 이 코드는 겉으로 보면 단순한 조회 + 변환처럼 보입니다.
- 학생이 먼저 봐야 할 포인트는 **어디서 연관 데이터에 접근하는지**입니다.

### 3. 내부에서 실제로 무슨 일이 벌어지는가

```sql
-- 1. 먼저 목록을 한 번 조회합니다.
select * from [main_table];

-- 2. 그다음 각 행마다 연관 데이터를 다시 읽을 수 있습니다.
select * from [related_table] where [foreign_key] = ?;
select * from [related_table] where [foreign_key] = ?;
select * from [related_table] where [foreign_key] = ?;
```

- 코드 한 블록은 짧아 보여도, DB 쪽에서는 쿼리가 여러 번 나갈 수 있습니다.
- 이 섹션에서는 **코드 1번 실행 = 쿼리 몇 번 실행인지**를 문장으로 풀어줍니다.

### 4. 그래서 왜 느려지거나 어려워지는가

- 쿼리 횟수가 늘어나면 DB가 더 많은 일을 하게 됩니다.
- 서버와 DB 사이 왕복도 그만큼 늘어납니다.
- 데이터가 5개일 때는 티가 약해도, 100개 1000개로 커지면 응답 시간이 급격히 늘 수 있습니다.
- 운영 중에는 로그가 많아지고, 어디서 느려졌는지 찾기도 더 어려워집니다.

이 부분은 반드시 **규모가 커질수록 무엇이 달라지는지**까지 설명합니다.

### 5. 대표 해결 코드는 어떻게 생기는가

```kotlin
@Query(
    "select distinct item from Item item " +
    "join fetch item.[relatedValues]"
)
fun findAllWithRelated(): List<Item>
```

- 이 코드는 문제를 완전히 다 해결하는 만능 답안이 아니라,
  **실무에서 문제를 줄일 때 어떤 방향으로 코드를 바꾸는지** 보여주는 예시입니다.
- 여기서는 “무엇을 한 번에 같이 읽어오려고 하는가”를 먼저 설명합니다.
- 가능하면 문제 코드와 해결 코드를 나란히 놓고 비교합니다.

### 6. 이번 시퀀스에서 어디까지 다루는가

- [이번 시퀀스에서 직접 구현하는 범위]
- [이번 시퀀스에서는 이름만 소개하는 해결책]
- [다음 시퀀스나 심화 주제로 넘기는 내용]

예시 문장:
- 이번 시퀀스에서는 문제를 인식하고 코드에서 찾는 데 집중합니다.
- 해결책은 `[fetch join / custom validator / role rule / cache evict]` 같은 이름만 먼저 익힙니다.
- 지금 단계에서는 “왜 이 문제가 생기는지 설명할 수 있는가”가 더 중요합니다.

## 코드 구조와 연결해서 보기

> 이 섹션은 **“각 코드가 무슨 일을 맡고 있는가”**를 기억하기 쉽게 정리하는 용도입니다.

### Controller는 무엇을 할까?

- 요청을 가장 먼저 받는 역할입니다.
- 어떤 URL과 어떤 HTTP 메서드가 연결되는지 여기서 확인할 수 있습니다.
- 보통 `@GetMapping`, `@PostMapping` 같은 어노테이션과 함께 등장합니다.
- 이번 프로젝트에서는 `[Controller 이름]`이 이 역할을 맡습니다.

> 기억 포인트  
> Controller는 “요청의 입구”입니다.

---

### Service는 무엇을 할까?

- 실제 처리 흐름을 모으는 역할입니다.
- Controller에 모든 로직을 넣지 않고 분리하는 이유도 여기에 있습니다.
- 저장, 조회, 검증, 변환 같은 핵심 처리가 여기서 일어납니다.
- 이번 프로젝트에서는 `[Service 이름]`이 이 역할을 맡습니다.

> 기억 포인트  
> Service는 “처리의 중심”입니다.

---

### Repository 또는 핵심 인프라 요소는 무엇을 할까?

- DB 접근, 보안, 캐시, 메시지 브로커, 외부 API 호출 등 바깥 기능과 연결되는 역할입니다.
- 이번 실습에서 어떤 기술이 핵심인지에 따라 이 자리에 들어오는 요소가 달라집니다.
- 예를 들어 JPA 실습에서는 Repository, Security 실습에서는 Filter/Config, Redis 실습에서는 RedisTemplate 또는 Cache 설정이 핵심일 수 있습니다.
- 이번 프로젝트에서는 `[Repository/Config/Filter/Template 이름]`이 이 역할을 맡습니다.

> 기억 포인트  
> 이 계층은 “애플리케이션 바깥과 연결되는 지점”입니다.

---

### DTO는 왜 필요할까?

- 요청과 응답을 분리하기 위해 사용합니다.
- Entity를 그대로 주고받으면 구조가 쉽게 섞이고, 나중에 수정도 어려워질 수 있습니다.
- DTO를 사용하면 지금 필요한 값만 안전하게 주고받을 수 있습니다.
- 이번 프로젝트에서는 `[Request DTO 이름]`, `[Response DTO 이름]`이 이 역할을 맡습니다.

> 기억 포인트  
> DTO는 “주고받는 데이터 전용 상자”입니다.

---

## 이번 실습에서 꼭 보면 좋은 포인트

- 어떤 코드가 이번 실습의 **시작점**인지
- 비즈니스 로직이 어느 **메서드**에 모여 있는지
- DB/설정/보안과 연결되는 지점이 어디인지
- 실제 실행 결과와 바로 연결되는 코드가 무엇인지
- 주석을 읽지 않아도 **흐름을 말로 설명할 수 있는지**

---

## 자주 헷갈리는 포인트

> FAQ처럼 길게 답하기보다, **학생이 실제로 많이 헷갈리는 실수**를 먼저 정리합니다.

- Controller가 모든 일을 하는 곳은 아닙니다.
- Service는 단순히 “중간에 있는 클래스”가 아니라, 처리 흐름을 모으는 핵심 계층입니다.
- Repository는 비즈니스 로직을 처리하는 곳이 아니라, DB 접근을 맡는 곳입니다.
- DTO는 귀찮아서 만드는 구조가 아니라, 요청/응답을 분리하기 위한 장치입니다.
- Validation은 잘못된 입력을 최대한 초반에 막기 위한 장치입니다.
- Exception Handling은 에러를 없애는 것이 아니라, **일관된 방식으로 응답하게 만드는 것**입니다.

---

## 한 줄 상황 예시로 감 잡기

> 너무 추상적으로 느껴질 때는, 실제 요청 상황으로 바꿔서 생각하면 이해가 쉬워집니다.

예시:
> 사용자가 회원가입 요청을 보냈는데 이메일이 비어 있다면,  
> 서버 안쪽까지 들어가기 전에 Validation이 먼저 잘못된 요청을 막아줍니다.

또는

> 사용자가 글 작성 요청을 보내면,  
> Controller가 요청을 받고 → Service가 글을 만들고 → Repository가 저장한 뒤 → 응답을 돌려줍니다.
