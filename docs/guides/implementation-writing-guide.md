# 구현 문서 작성 가이드

## 문서 목적

이 문서는 Codex가 실습용 구현 문서를 작성할 때 따라야 하는 작성 기준을 정리합니다.

구현 문서의 목적은 이론 설명이 아니라,
학생이 손을 움직이는 순서를 보이게 만드는 것입니다.

---

## 핵심 원칙

### 1. 결과보다 순서를 먼저 씁니다

좋지 않은 방식:

- Service를 만든다
- Repository를 만든다

좋은 방식:

1. Request DTO를 만든다.
2. 저장할 데이터를 만든다.
3. 저장소를 만든다.
4. Service에서 저장소를 호출한다.
5. Controller를 연결한다.
6. Swagger 또는 테스트로 결과를 확인한다.

### 2. 제목은 도메인 중심으로 씁니다

좋지 않은 제목:

- Week 13
- Week 14

좋은 제목:

- 요청-응답과 메모리 CRUD
- 영속성 저장과 계층 분리
- 안전한 요청 처리

### 3. 학생이 직접 구현할 것과 제공할 것을 분리합니다

문서에는 반드시 아래 두 구간이 분리되어 있어야 합니다.

- 학생이 직접 구현할 순서
- 미리 제공할 것

### 4. 학생이 바로 실행할 수 있게 적습니다

좋은 구현 문서는 아래 질문에 바로 답할 수 있어야 합니다.

- 지금 무엇부터 만들지?
- 어느 파일을 열지?
- 어디까지 직접 구현하지?
- 무엇으로 확인하지?

---

## 권장 문서 구조

각 구현 문서는 아래 순서를 기본 구조로 사용합니다.

1. 이 도메인이 필요한 이유
2. 학생이 완성할 최종 흐름
3. 학생이 직접 구현할 순서
4. TODO를 넣을 파일
5. 미리 제공할 것
6. 학생 체크리스트
7. 강사 / PPT 체크리스트
8. 다음 도메인 연결 포인트

---

## TODO 작성 규칙

### 1. TODO는 큰 제목이 아니라 단계형 지시여야 합니다

좋지 않은 예시:

```kotlin
// TODO: 예외 처리 구현
```

좋은 예시:

```kotlin
// TODO 1. 요청 DTO의 title 값이 비어 있는지 확인하세요.
// TODO 2. 비어 있다면 BadRequestException을 던지세요.
// TODO 3. 예외 메시지를 ErrorResponse로 반환하세요.
```

### 2. TODO는 한 파일에서 너무 많지 않게 둡니다

권장 기준:

- 한 주제당 핵심 TODO 3~7개
- 한 파일에 지나치게 몰아넣지 않기
- 흐름이 끊기지 않게 배치하기

### 3. TODO는 확인 방법과 연결되어야 합니다

가능하면 TODO는 아래 둘 중 하나와 연결되어야 합니다.

- Swagger 실행
- 테스트 실행

---

## 구현 범위 조절 기준

학생이 직접 구현할 것으로 적합한 항목:

- 핵심 요청/응답 흐름
- 저장/조회/검증/인증의 중심 로직
- 다음 단계에서도 재사용할 구조

미리 제공하는 편이 좋은 항목:

- 반복 설정
- 긴 설정 파일
- 외부 인프라 연결의 세부값
- 장황한 배포 스크립트
- 부가 기능 전부

---

## 최종 기준

좋은 구현 문서는 설명이 많은 문서가 아니라,
학생이 다음 손동작을 바로 결정할 수 있는 문서입니다.
