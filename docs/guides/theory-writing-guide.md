# 이론 문서 작성 가이드

## 문서 목적

이 문서는 A&I 백엔드 토픽 레포에서 이론 문서를 작성할 때 지켜야 할 기준을 정리한 가이드입니다.

이 문서의 목표는 단순합니다.

학생이 README에서 연결된 이론 문서를 읽고, 실습 내용을 어렵지 않게 다시 이해하고, 나중에 다시 봤을 때도 핵심 흐름을 쉽게 떠올릴 수 있게 만드는 것

즉, 이 문서는 백과사전식 설명을 만드는 가이드가 아닙니다.  
이론 문서를 읽고 나서 학생이 아래를 할 수 있어야 합니다.

- 왜 이 기술을 배우는지 설명할 수 있다.
- 이번 실습의 흐름을 순서대로 말할 수 있다.
- 중요한 코드가 어디인지 다시 찾을 수 있다.
- 다음 주차와 어떻게 연결되는지 떠올릴 수 있다.

한마디로, 이론 문서는 정보 정리 문서가 아니라  
이해 + 회상 + 자기 설명을 돕는 학습 문서여야 합니다.

---

## 이 가이드가 지향하는 문서

좋은 이론 문서는 아래 특징을 가집니다.

- 정의부터 시작하지 않고, 왜 배우는지부터 보여줍니다.
- 개념 설명만 하지 않고, 반드시 코드와 연결합니다.
- 문제 상황이 코드에서 어떻게 나타나는지 보여줍니다.
- 길게 다 설명하려 하지 않고, 기억에 남을 핵심만 남깁니다.
- 학생이 다시 봤을 때 “아, 이거 이런 흐름이었지”가 떠오르게 만듭니다.
- 읽는 문서에서 끝나지 않고, 스스로 설명해보게 만듭니다.

---

## 핵심 작성 원칙

### 0. 이론 문서는 반드시 3층 구조를 유지합니다

실무 확장 개념이 붙는 시퀀스에서도,
이론 문서는 아래 3층을 모두 가져야 합니다.

1. `기초 이론`
2. `현재 코드 흐름 설명`
3. `실무 확장 개념`

실무 확장 개념은 기본 설명을 대체하는 것이 아니라
기초 설명 위에 얹히는 마지막 층입니다.

좋은 구조 예:

- 왜 배우는가
- 기본 개념과 핵심 용어
- 현재 코드 흐름 한눈에 보기
- 중요한 코드 먼저 보기
- 실무 확장 개념
- 직접 말해보기 / 복습 체크리스트

나쁜 구조 예:

- 실무 문제 상황
- 실무 위험
- 해결 코드

만 길게 있고,

- 기술이 기본적으로 무엇인지
- 현재 코드에서 어떤 클래스가 어떤 역할을 하는지

같은 설명이 사라진 문서

이 경우 문서는 "실무 팁 메모"는 될 수 있어도,
"학생용 이론 문서"는 될 수 없습니다.

또한 현재 시퀀스 문서를 쓸 때는
바로 이전 시퀀스 문서를 함께 보고,
이번 시퀀스 이해에 여전히 필요한 기초 용어와 흐름이 무엇인지 먼저 확인해야 합니다.

즉, 이전 시퀀스 문서에 설명이 있다고 해서
현재 시퀀스 문서에서 그 설명을 비워두면 안 됩니다.

### 1. 정의보다 문제 상황부터 시작합니다

도입부는 기술 이름이나 정의부터 시작하지 않습니다.  
먼저 “왜 이걸 배우는가”를 열어야 합니다.

좋은 예:

- 메모리에만 저장하면 서버를 껐다 켤 때 데이터가 사라집니다.
- 사용자를 구분하지 못하면 로그인 기능이 없는 서비스와 큰 차이가 없습니다.
- 매번 DB만 조회하면 응답 속도가 느려질 수 있습니다.

아쉬운 예:

- JPA는 Java Persistence API의 약자입니다.
- JWT는 JSON Web Token입니다.

정의는 그 다음에 나와도 늦지 않습니다.  
먼저 학생이 문제 상황을 이해해야 개념이 머리에 남습니다.

---

### 2. 이론 문서는 “정리용”이 아니라 “회상용”이어야 합니다

학생은 문장을 그대로 외우지 않습니다.  
대신 아래처럼 장면으로 기억합니다.

- 아, 이 주차는 요청이 들어와서 DB까지 가는 흐름이었지
- 아, 이 주차는 `@Valid` 붙여서 입력값을 막았지
- 아, 이 주차는 JWT로 로그인 이후 요청을 구분했지

그래서 문서는 설명만 잘 되어 있으면 충분하지 않습니다.  
다시 봤을 때 기억이 빨리 살아나는 구조여야 합니다.

이를 위해 각 문서에는 아래 요소가 들어가는 것을 권장합니다.

- 이번 주차 한 줄 요약
- 먼저 이것만 기억해도 되는 핵심 3개
- 실습 흐름 한눈에 보기
- 중요한 코드 먼저 보기
- 직접 말해보기
- 복습 체크리스트

---

### 3. 용어는 이번 실습에 필요한 만큼만 설명합니다

용어를 길게 백과사전처럼 설명하지 않습니다.  
이번 실습을 이해하는 데 필요한 만큼만 설명합니다.

좋은 예:

`Entity`  
DB 테이블과 연결되는 객체입니다.  
이번 실습에서는 `Idol` 클래스가 그 역할을 맡습니다.

이 정도면 입문자 문서에서는 충분한 경우가 많습니다.

좋은 용어 설명은 아래 네 가지를 함께 담습니다.

- 뜻
- 왜 필요한가
- 이번 코드에서는 어디에 보이는가
- 짧은 상황 예시

주의:

- 실무 확장 개념을 넣는다고 해서 용어 설명을 빼면 안 됩니다.
- 학생이 기술을 처음 본다고 가정한다면, 기초 용어 설명이 먼저 살아 있어야 합니다.
- `implementation.md`가 알아서 설명해줄 것이라고 넘기지 않습니다.
- `theory.md` 안에서 기본 개념을 먼저 이해할 수 있어야 합니다.
- 이전 시퀀스에서 이미 설명한 용어라도, 이번 시퀀스 이해에 필수면 다시 설명합니다.
- 단, 이전 시퀀스 문장을 그대로 복사하는 것이 아니라 이번 맥락에 맞게 다시 씁니다.

---

### 4. 반드시 코드와 연결합니다

이론 문서는 개념만 말하면 금방 붕 뜹니다.  
반드시 아래 질문에 답해야 합니다.

- 그래서 이 개념이 코드에서 어디에 보이는가?
- 이번 실습에서 어떤 코드가 그 역할을 하는가?
- 학생이 가장 먼저 봐야 할 줄은 어디인가?

좋은 예:

- Repository는 DB 접근 역할을 합니다.
- 이번 코드에서는 `IdolRepository`가 그 역할을 맡습니다.
- `IdolService.create(...)`에서 `idolRepository.save(...)`를 호출하는 부분을 보면 저장 흐름이 보입니다.

여기서 중요한 것은 파일 경로만 나열하지 않는 것입니다.  
파일 경로는 보조 정보이고, 핵심은 코드와 역할을 연결해 설명하는 것입니다.

또 하나 중요한 점은,
이 코드 연결이 실무 확장 개념 설명만을 위한 것이어서는 안 된다는 점입니다.

좋은 이론 문서는 먼저 아래를 설명합니다.

- 현재 요청이 어디서 시작하는가
- 어떤 클래스가 어떤 역할을 맡는가
- 핵심 메서드가 어디에 있는가

그 다음에야

- 이 코드가 실무에서 어떤 문제를 만들 수 있는가
- 어떻게 보강할 수 있는가

로 넘어갑니다.

---

### 4-1. 실무 확장 개념은 "문제 코드 분석"으로 설명합니다

실무 확장 개념이 붙는 시퀀스에서는
단순히 “이런 문제가 있습니다”라고 쓰면 학생이 공감하기 어렵습니다.

반드시 아래 질문에 답해야 합니다.

- 어떤 코드가 처음에는 자연스럽게 보이는가?
- 그 코드는 내부에서 어떤 쿼리나 동작을 추가로 만들까?
- 그래서 왜 속도가 느려지거나 운영이 어려워질까?
- 데이터가 많아지면 무엇이 달라질까?
- 이 문제를 줄이기 위한 대표 해결 코드는 어떤 모양일까?

좋은 예:

- 게시글 목록을 가져온 뒤 연관된 댓글을 각 게시글마다 다시 읽는 코드
- `게시글 1번 조회 + 댓글 N번 조회`가 되는 쿼리 흐름
- 로컬에서는 괜찮아 보여도 데이터가 100개, 1000개가 되면 왜 느려지는지 설명
- 그리고 `fetch join` 같은 해결 코드가 어떤 식으로 생겼는지 보여주기

핵심은 개념 설명 -> 문제 코드 -> 내부 동작 -> 실무 영향 -> 해결 코드 예시 순서가 보여야 한다는 점입니다.

여기서 말하는 `개념 설명`은
실무 확장 개념 자체만이 아니라,
이번 시퀀스의 기본 기술 설명도 포함합니다.

예를 들어 `05`라면,

- 먼저 OAuth2와 SMTP의 기본 역할을 설명하고
- 그 다음에 계정 연결 정책과 계정 존재 여부 노출 위험을 붙입니다

즉, 실무 확장 개념은 언제나 기본 개념을 설명한 뒤에 나와야 합니다.

---

### 5. 중요한 코드는 짧게 보여주고, 주석으로 이해를 돕습니다

학생은 긴 코드보다 짧은 핵심 코드에서 흐름을 먼저 잡습니다.  
그래서 코드 블록은 짧게 자르고, 이해에 도움이 되는 주석을 함께 넣습니다.

좋은 예:

```kotlin
@GetMapping
fun getAll(): List<IdolResponse> {
    // 요청을 받는 시작점은 Controller입니다.
    return idolService.getAll()
    // 실제 목록 조회는 Service가 맡습니다.
}
```

이런 주석은 문법 설명보다 역할 설명에 가까워야 합니다.

좋은 주석:
- 요청을 받는 시작점입니다.
- 실제 처리는 Service가 맡습니다.
- 이 줄에서 DB 저장이 일어납니다.
- 응답용 DTO로 다시 감싸서 돌려줍니다.

아쉬운 주석:
- 함수입니다.
- 변수입니다.
- Kotlin 문법 설명만 길게 적는 주석

코드 블록 안의 주석은 학생이 흐름을 따라가게 돕는 용도여야 합니다.

실무 확장 개념이 들어갈 때는
아래 형태의 코드 블록도 적극적으로 사용하는 것이 좋습니다.

- 문제를 만드는 예시 코드
- 예상되는 쿼리 흐름
- “왜 여기서 N번 더 호출되는가” 같은 해설 주석
- 대표 해결 방법을 보여주는 예시 코드
- “이 코드가 왜 쿼리 수를 줄이는가” 같은 비교 설명

즉, 좋은 이론 문서의 코드 블록은
정상 흐름 코드뿐 아니라 문제가 생기는 코드와 해결 코드도 분석 대상으로 보여줘야 합니다.

---

### 6. 설명체보다 안내체에 가깝게 씁니다

학생이 문서를 읽을 때 “평가받는 느낌”보다  
“옆에서 같이 정리해주는 느낌”이 들어야 합니다.

권장 톤:

- 쉬운 문장
- 짧은 문단
- 부드러운 존댓말
- 실습 감각이 느껴지는 표현
- 추상적 정의보다 연결 설명
- “즉”, “쉽게 말하면”, “이번 코드에서는”, “여기서는” 같은 연결 표현 사용

아쉬운 예:

- Controller는 요청을 받는 곳이다.
- Repository는 DB 접근 역할을 한다.

권장 예:

- Controller는 요청을 받는 곳입니다.
- Repository는 DB 접근 역할을 해요.
- 이번 코드에서는 `IdolRepository`가 그 역할을 맡습니다.

---

### 7. 한 문서에서 핵심 개념을 너무 많이 다루지 않습니다

문서가 친절하다고 해서 많은 내용을 넣으면 오히려 기억이 흐려집니다.  
한 문서에서 학생이 확실히 붙잡아야 할 개념은 많아야 3개 정도가 적당합니다.

예:

- CRUD 주차  
  - 요청/응답 흐름
  - 계층 분리
  - DB 저장/조회

- Validation 주차  
  - 입력값 검증
  - 실패 응답
  - 전역 예외 처리

핵심은 많이 담는 것보다, 다시 떠올릴 수 있게 담는 것입니다.

실무 확장 개념도 마찬가지입니다.
한 시퀀스에 여러 실무 이슈를 넣기보다,
대표 문제 하나를 코드 기준으로 깊이 있게 설명하는 편이 더 좋습니다.

## 문서가 반드시 해줘야 하는 일

좋은 이론 문서는 아래 네 가지를 해줘야 합니다.

### 1. 왜 배우는지 먼저 떠오르게 해줘야 합니다
학생이 “왜 이걸 배우지?”를 놓치면 뒤 내용이 다 흩어집니다.

### 2. 전체 흐름을 먼저 보여줘야 합니다
세부 설명 전에 큰 흐름을 먼저 봐야 길을 잃지 않습니다.

### 3. 핵심 코드와 개념을 강하게 연결해줘야 합니다
개념이 어디에 보이는지 바로 찾을 수 있어야 합니다.

### 4. 학생이 직접 말해보게 만들어야 합니다
읽고 이해한 것 같아도, 직접 설명해봐야 진짜 이해했는지 드러납니다.

---

## 완료 전 최종 검증 질문

이론 문서를 완료했다고 보기 전에 아래를 반드시 확인합니다.

1. 이 문서에 `기초 개념 설명`이 남아 있는가
2. 이 문서가 `현재 코드 흐름`을 설명하는가
3. 이 문서가 `실무 확장 개념`을 별도 층으로 설명하는가
4. 실무 확장 개념이 기본 설명을 밀어내고 있지는 않은가
5. 이전 시퀀스 문서에 있던 필수 용어가 이번 문서에서 빈칸처럼 빠지지 않았는가
6. 학생이 이 문서만 읽어도 기술을 처음 이해할 수 있는가

이 질문 중 하나라도 `아니오`라면,
문서는 다시 써야 합니다.

### 5. 실무 확장 개념은 왜 느려지고 왜 어려워지는지까지 설명해줘야 합니다
특히 성능, 보안, 운영 개념은 “문제가 있다”에서 끝내지 말고,
실제로 어떤 비용이 늘어나는지까지 보여줘야 학생이 공감할 수 있습니다.

## 권장 문서 구조

### 필수 섹션

### 8. 중요한 코드 먼저 보기

이 섹션은 필수입니다.

가장 먼저 봐야 할 코드를 짧은 코드 블록으로 보여줍니다.  
권장 방식은 아래와 같습니다.

- 코드 블록은 5~15줄 정도로 자릅니다.
- 블록은 2~4개 정도면 충분합니다.
- 블록마다 왜 중요한지 1~2문장으로 적습니다.
- 가능하면 코드 블록 안에 짧은 설명 주석을 넣습니다.
- 파일 경로는 보조 정보로 짧게 적습니다.

예시:

```kotlin
@PostMapping
fun create(@RequestBody request: IdolCreateRequest): IdolResponse {
    // 요청이 처음 들어오는 곳입니다.
    return idolService.create(request)
    // 실제 생성 처리는 Service에게 맡깁니다.
}
```

좋은 코드 블록 설명 예:

- 이 코드는 요청의 시작점을 보여줍니다.
- 여기서는 `idolService.create(request)`를 먼저 보세요.
- Controller가 직접 처리하지 않고 역할을 넘긴다는 점이 중요합니다.
- 파일: `IdolController.kt`

### 9. 핵심 용어를 쉬운 말로 정리하기

- 뜻
- 왜 필요한가
- 이번 코드에서는 어디에 보이는가
- 짧은 상황 예시

예시:

`DTO`

- 뜻  
  요청과 응답에 필요한 데이터만 따로 담는 객체입니다.
- 왜 필요한가

- 짧은 상황 예시  
  사용자가 글을 작성할 때 제목과 내용만 보내면 되기 때문에, 요청 DTO에는 필요한 값만 담습니다.

## 코드 블록 작성 규칙

코드 블록은 학생 기억을 돕는 핵심 요소입니다.  
아래 규칙을 기본으로 합니다.

### 1. 코드 블록은 짧게 자릅니다

- 한 블록은 5~15줄 정도를 권장합니다.
- 긴 코드를 통째로 붙이지 않습니다.
- 지금 설명하는 개념이 보이는 부분만 가져옵니다.

### 2. 코드 블록 안의 주석은 역할 설명 위주로 씁니다

좋은 예:

```kotlin
val saved = idolRepository.save(entity)
// 이 줄에서 실제 DB 저장이 일어납니다.
```

아쉬운 예:

```kotlin
val saved = idolRepository.save(entity)
// save 메서드를 호출합니다.
```

좋은 주석은 “무슨 문법인가”보다  
“이 줄이 전체 흐름에서 무슨 역할인가”를 알려줍니다.

### 3. 주석은 너무 많이 넣지 않습니다

주석이 많아지면 오히려 코드가 흐려집니다.  
한 블록당 2~4개 정도면 충분합니다.

### 4. 실제 동작과 다른 설명을 넣지 않습니다

설명용 주석을 달아도 괜찮지만,  
학생에게 잘못된 동작을 상상하게 만드는 주석은 피합니다.

### 5. 설명은 코드 블록 아래에서도 한 번 더 정리합니다

코드 주석만으로 끝내지 말고, 블록 아래에 아래를 짧게 적습니다.

- 이 코드를 왜 먼저 봐야 하는지
- 학생이 어떤 줄을 먼저 보면 되는지
- 이 코드가 어떤 흐름을 보여주는지
- 파일 경로

## 주제별 적용 기준

### CRUD / JPA / DTO / Validation / Exception / JWT 같은 주제
아래 구조가 잘 맞습니다.

- 요청/응답 흐름
- 역할 분리
- 핵심 코드
- 계층 구조
- 자주 헷갈리는 포인트

### Docker / AWS / CI/CD / Redis / MQ / WebSocket 같은 주제
아래 구조를 중심으로 설명합니다.

- 왜 이 기술이 필요한가
- 실습 흐름
- 핵심 설정 또는 인프라 코드
- 어디서 동작이 갈리는가
- 운영 또는 구조 관점에서 왜 중요한가

즉, 모든 주제를 Controller/Service/DTO 구조에 억지로 맞추지 않습니다.

- 파일 경로만 나열하는 문서
- 한 문서에 너무 많은 개념을 넣는 문서
- 학생이 다시 봤을 때 핵심이 바로 안 보이는 문서
- 주석 없는 긴 코드 블록만 여러 개 나열한 문서
- 다음 주차와 연결이 끊긴 문서
- 공식 문서 내용을 거의 그대로 옮겨 적는 문서

## 참고 자료 규칙

참고 자료는 아래 우선순위를 권장합니다.

1. 공식 문서
2. 공식 가이드 또는 공식 블로그
3. 보조 학습 자료

가능하면 참고 자료는 문서 맨 아래에 짧게 정리합니다.  
자료를 많이 넣는 것보다, 믿을 수 있는 자료를 적게 넣는 편이 좋습니다.

---

## 최종 체크리스트

문서 작성이 끝나면 아래를 확인합니다.

- [ ] 제목만 보고도 무엇을 배우는 문서인지 알 수 있는가
- [ ] 도입부가 정의가 아니라 문제 상황부터 시작하는가
- [ ] 이번 주차 한 줄 요약이 있는가
- [ ] 먼저 기억해야 할 핵심 3개가 보이는가
- [ ] 실습 흐름을 순서대로 보여주는가
- [ ] 중요한 코드 블록이 짧게 들어가 있는가
- [ ] 코드 블록 안에 이해를 돕는 주석이 있는가
- [ ] 개념 설명이 반드시 코드와 연결되는가
- [ ] 학생이 직접 말해볼 질문이 들어가 있는가
- [ ] 복습 체크리스트가 있는가
- [ ] 오늘 꼭 기억할 핵심 3줄이 있는가
- [ ] 다음 주차와 연결되는 문장이 있는가
- [ ] 공식 문서 중심의 참고 자료가 있는가

## 앞으로의 적용 규칙

앞으로 만드는 모든 토픽 레포의 이론 문서는 이 가이드를 따릅니다.

적용 기준은 아래와 같습니다.

1. README에는 실습 안내와 진입 흐름을 중심으로 적습니다.
2. 용어, 개념, 상세 설명은 `docs/*.md`로 분리합니다.
3. README에서 이론 문서로 반드시 연결합니다.
4. 이론 문서는 이 가이드의 구조와 문체 규칙을 따릅니다.
5. 각 문서는 학생이 “이해한 뒤 다시 떠올릴 수 있는가”를 기준으로 검토합니다.
