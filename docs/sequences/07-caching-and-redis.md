## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스들에서 만든 조회 흐름 위에,
학생이 아래 흐름을 직접 연결하면서 **캐시가 왜 필요한지**를 이해하게 만드는 것입니다.

- 먼저 캐시를 조회한다
- 캐시에 없으면 DB를 조회한다
- DB 조회 결과를 캐시에 저장한다
- 다음 조회에서는 캐시 hit가 일어난다
- TTL을 통해 캐시가 영구 저장소가 아니라는 점을 이해한다

즉, 이번 시퀀스는 Redis 전체를 깊게 배우는 단계가 아닙니다.
학생이 **DB와 캐시의 역할 차이**, **hit/miss 흐름**, **왜 자주 조회되는 데이터에 캐시를 붙이는지**
입문 수준에서 이해하게 만드는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. DB와 캐시의 차이를 설명할 수 있다.
2. 캐시 hit와 miss를 설명할 수 있다.
3. 조회 시 캐시를 먼저 보고, 없으면 DB를 본다는 흐름을 말할 수 있다.
4. TTL이 왜 필요한지 설명할 수 있다.
5. 어떤 데이터에 캐시를 붙이면 좋은지 입문 수준에서 말할 수 있다.
6. 같은 요청을 두 번 보냈을 때 hit/miss 차이를 확인할 수 있다.

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: Redis 캐시 구현
````

좋은 예:

```kotlin id="6c5j4d"
// TODO 1. 먼저 캐시에서 postId에 해당하는 값을 조회하세요.
// TODO 2. 캐시에 값이 없으면 repository에서 조회하세요.
// TODO 3. 조회한 값을 Redis에 저장하세요.
// TODO 4. 다음 조회부터는 캐시 hit가 일어나는지 확인하세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* hit/miss 차이를 직접 느끼게 유도한다.

예:

```kotlin id="gh9dj8"
// TODO: 캐시가 없다고 바로 실패 처리하지 마세요.
// miss는 정상적인 상황이며, 이때는 DB 조회로 자연스럽게 이어져야 합니다.
```

또는

```kotlin id="80cxt7"
// TODO: Redis를 영구 저장소처럼 설명하지 마세요.
// TTL이 지나면 다시 DB 조회가 일어날 수 있어야 합니다.
```
