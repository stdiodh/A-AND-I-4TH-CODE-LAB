# 11. 리팩토링과 기초 보강

## 목표

이미 만든 기능의 동작을 보존하면서 Service의 입력 정리, 조회, 검증, 변환 책임을 작은 helper로 분리합니다.
새 기능을 크게 추가하는 단계가 아니라 책임 분리, 구조 개선, 테스트 기반 검증에 집중합니다.

## 이 시퀀스에서 배우는 것

- 리팩토링은 동작을 바꾸지 않고 구조를 개선하는 작업이라는 점
- 큰 패키지 이동과 작은 책임 분리의 차이
- 입력 검증, 조회, 변환, 예외 처리 경계
- `PostService`, `AuthService`의 책임 분리 기준
- 리팩토링 범위를 테스트가 보호하는 방식
- 테스트로 동작 보존 확인

2. `PostService`의 요청 정리, 게시글 조회, 작성자 검증 후보를 찾습니다.
3. `AuthService`의 사용자 조회와 토큰 응답 생성 후보를 찾습니다.
4. 반복되는 흐름을 private helper로 작게 추출합니다.
5. 잘못된 입력과 없는 사용자에 대한 예외 응답을 보강합니다.
6. 테스트 이름이 보호하는 동작과 경계를 확인합니다.
7. 테스트를 다시 실행해 API 동작이 유지되는지 확인합니다.
8. feature-based 패키지 전환은 이번 답안에 포함되지 않은 후속 선택지로 비교합니다.

## 실행 방법

```bash
docker compose up -d
./gradlew bootRun
```

## 테스트 방법

```bash
./gradlew test
```

리팩토링 전후에 같은 명령을 각각 실행합니다.

테스트가 확인하는 것:

- 리팩토링 전 현재 API 동작을 고정합니다.
- helper 추출과 예외 보강 후에도 같은 테스트가 통과하는지 확인합니다.
- Service 책임 분리가 외부 API 동작을 바꾸지 않았는지 확인합니다.

실패하면 먼저 볼 것:

- helper 추출 직후라면 인자와 반환값, 예외 타입이 유지되는지 먼저 확인합니다.
- 책임 분리 후 실패했다면 변경 전후 테스트 이름과 실패 케이스를 비교합니다.
- 새 구조를 만들면서 API path, response body, status code를 바꾸지 않았는지 봅니다.

완료 기준:

- 리팩토링 전 `./gradlew test`가 통과합니다.
- 리팩토링 후 `./gradlew test`가 다시 통과합니다.
- API 동작 보존과 작은 책임 분리를 테스트 결과로 설명할 수 있습니다.

## 확인할 API 또는 화면

- 리팩토링 전후 테스트 결과
- 변경 전후 `PostService`, `AuthService` 메서드 흐름
- 추출한 helper가 입력 정리, 조회, 검증, 변환 중 한 책임만 갖는지 확인
- feature-based 패키지 이동이 필요해지는 규모와 기준 토론

## 자주 발생하는 문제

- 리팩토링 중 기능을 함께 바꿉니다. 이번 목표는 동작 보존입니다.
- 큰 패키지 이동과 책임 분리를 한 번에 섞습니다. 이번 단계에서는 테스트로 확인 가능한 작은 책임 분리만 수행합니다.
- 테스트 없이 구조를 바꿉니다. 변경 전후 테스트 결과를 비교합니다.
- 과도한 구조 용어를 외우는 데 집중합니다. 핵심은 변경할 때 볼 파일을 가까이 모으는 것입니다.

## 완료 기준

- 리팩토링 전후 주요 기능 동작이 유지됩니다.
- Service의 반복 흐름이 작은 helper로 분리되었습니다.
- 입력 검증과 비즈니스 조회 실패가 구분됩니다.
- 테스트가 리팩토링 안전망 역할을 합니다.
- `./gradlew test`가 통과합니다.
