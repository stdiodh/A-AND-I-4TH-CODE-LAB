당신은 A&I 백엔드 커리큘럼용 실습 레포와 문서를 설계하는 Codex입니다.

지금부터는 전체 커리큘럼 중 **11. 리팩토링과 기초 보강** 시퀀스만 다룹니다.
다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 11 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스들에서 만든 기능 위에,
학생이 아래 흐름을 직접 연결하면서 **좋은 구조가 무엇인지**, **왜 기초를 다시 묶는 시간이 중요한지**를 이해하게 만드는 것입니다.

- 지금까지 만든 코드를 다시 읽는다
- 역할이 섞인 부분을 찾는다
- Validation 또는 Exception Handling을 보강한다
- 테스트를 추가한다
- README 또는 문서를 보강한다

즉, 이번 시퀀스는 새로운 무거운 기술을 하나 더 배우는 단계가 아닙니다.
학생이 **지금까지 배운 요청/저장/검증/인증/테스트 흐름이 하나로 연결된다는 점**을 보고,
코드를 더 읽기 쉽고 점검하기 쉬운 형태로 다듬는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. 좋은 구조가 왜 읽기 쉬운지 설명할 수 있다.
2. 역할이 섞인 코드와 분리된 코드를 비교할 수 있다.
3. Validation, Exception Handling, 테스트가 따로 노는 주제가 아니라 연결된다는 점을 설명할 수 있다.
4. 기존 Service를 더 읽기 쉽게 정리할 수 있다.
5. 보강이 필요한 테스트를 추가할 수 있다.
6. 지금까지 배운 흐름을 한 번에 설명할 수 있다.

## 9. 코드 설계 규칙

이번 시퀀스의 코드는 아래 원칙을 따라야 합니다.

- Kotlin + Spring Boot 기준
- 기존 코드 위에서 개선이 일어나야 함
- Service 하나를 중심으로 리팩토링 포인트가 보이게 유지
- Validation 또는 Exception Handling 보강은 1개 안팎의 대표 흐름만 다룸
- 테스트는 1~2개 정도 추가하는 수준으로 유지
- README 또는 문서 보강은 실제로 학생이 다시 보기 쉬운 형태가 되게 함
- 코드 길이는 짧고 “전/후 비교”가 잘 보이게 유지

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: 리팩토링
````

좋은 예:

```kotlin id="h9w9n5"
// TODO 1. 이 메서드에서 요청 검증, 저장, 응답 변환이 한 번에 섞여 있는지 확인하세요.
// TODO 2. 지금 단계에서 가장 먼저 분리할 책임이 무엇인지 정하세요.
// TODO 3. 기능은 유지한 채로 읽기 쉬운 흐름으로 정리하세요.
```

테스트 보강 예:

```kotlin id="h92az9"
// TODO 1. 현재 이 흐름을 검증하는 테스트가 있는지 먼저 확인하세요.
// TODO 2. 없다면 정상 케이스 또는 실패 케이스 중 하나를 추가하세요.
// TODO 3. 리팩토링 후에도 테스트가 계속 통과하는지 확인하세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 무엇을 먼저 봐야 하는지 드러나게 적는다.
* “새 기능 추가”보다 “구조 정리”라는 목적이 드러나게 적는다.

예:

```kotlin id="f6erq6"
// TODO: 기능을 더 늘리려고 하지 마세요.
// 이번 단계의 핵심은 이미 있는 코드를 더 읽기 쉽고 점검하기 쉽게 만드는 것입니다.
```

또는

```text id="vjlwm2"
# TODO: README를 길게 늘리기보다,
# 학생이 다시 봤을 때 핵심 흐름이 빨리 떠오르도록 정리하세요.
```

## 14. starter 코드 파일 작성 규칙

starter 파일은 아래 성격을 가져야 합니다.

* 바로 읽고 수정할 수 있어야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* 전/후 차이를 비교할 수 있어야 함

필수 starter 파일 예시:

* `src/main/kotlin/.../service/PostService.kt`
* `src/main/kotlin/.../service/AuthService.kt`
* `src/main/kotlin/.../exception/GlobalExceptionHandler.kt`
* `src/test/kotlin/.../*Test.kt`
* `README.md`

필요하면 추가 가능:

* `ErrorResponse.kt`
* `docs/theory.md`

단, 범위를 과하게 넓히지 마세요.
