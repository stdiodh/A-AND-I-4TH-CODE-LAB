다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 08 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스들에서 만든 HTTP 기반 요청/응답 흐름 위에,
학생이 아래 흐름을 직접 연결하면서 **실시간 통신이 왜 필요한지**를 이해하게 만드는 것입니다.

- 클라이언트가 메시지를 보낸다
- 서버가 메시지를 받는다
- 서버가 특정 topic으로 다시 메시지를 보낸다
- 연결된 클라이언트가 실시간으로 메시지를 받는다

즉, 이번 시퀀스는 WebSocket 전체를 깊게 배우는 단계가 아닙니다.
학생이 **HTTP는 요청이 와야 응답할 수 있지만, 실시간 기능은 연결을 유지하며 서버가 다시 밀어줄 수 있다**는 감각을
입문 수준에서 이해하게 만드는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. HTTP와 WebSocket의 차이를 설명할 수 있다.
2. 서버가 클라이언트로 메시지를 다시 보낼 수 있는 이유를 말할 수 있다.
3. 메시지 DTO가 어떤 역할을 하는지 설명할 수 있다.
4. 서버가 받은 메시지를 topic으로 broadcast 하는 흐름을 말할 수 있다.
5. 테스트 페이지에서 메시지를 보내고 다시 받는 실시간 흐름을 확인할 수 있다.
6. 채팅/알림 같은 기능에서 왜 이런 구조가 필요한지 말할 수 있다.

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: 웹소켓 구현
````

좋은 예:

```kotlin
// TODO 1. 클라이언트가 보낸 메시지를 받으세요.
// TODO 2. 받은 메시지를 topic 경로로 다시 보내세요.
// TODO 3. 테스트 페이지에서 구독 중인 클라이언트가 메시지를 받는지 확인하세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* HTTP 요청/응답 흐름과 실시간 흐름 차이를 느끼게 유도한다.

예:

```kotlin
// TODO: 메시지를 DB에 저장하는 것까지 확장하지 마세요.
// 이번 단계의 목표는 '실시간으로 다시 보내는 흐름'을 보는 것입니다.
```

또는

```kotlin
// TODO: WebSocket 설정을 과하게 늘리지 마세요.
// 지금은 endpoint 1개와 topic 흐름 1개만 보여주면 충분합니다.
```

* 바로 실행은 가능해야 함

필수 starter 파일 예시:

* `src/main/kotlin/.../dto/ChatMessage.kt`
* `src/main/kotlin/.../controller/WebSocketController.kt`
* `src/main/kotlin/.../config/WebSocketConfig.kt`
* `src/main/resources/static/chat-test.html`

필요하면 추가 가능:

* `MessageResponse.kt`
* `WebSocketMessageService.kt`

단, 범위를 과하게 넓히지 마세요.
