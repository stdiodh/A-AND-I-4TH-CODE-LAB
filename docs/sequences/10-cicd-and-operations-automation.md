당신은 A&I 백엔드 커리큘럼용 실습 레포와 문서를 설계하는 Codex입니다.

지금부터는 전체 커리큘럼 중 **10. 자동화와 운영 흐름** 시퀀스만 다룹니다.
다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 10 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스에서 만든 배포 흐름 위에,
학생이 아래 흐름을 직접 연결하면서 **자동화가 왜 필요한지**를 이해하게 만드는 것입니다.

- 코드가 변경된다
- 빌드가 실행된다
- 테스트가 실행된다
- 배포가 실행된다
- 배포 후 확인 단계를 거친다

즉, 이번 시퀀스는 CI/CD 플랫폼 전체를 깊게 배우는 단계가 아닙니다.
학생이 **수동 배포는 반복하기 어렵고 실수하기 쉽다**는 점,
그리고 **자동화의 핵심은 속도보다 일관성**이라는 점을 입문 수준에서 이해하게 만드는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. 수동 배포와 자동 배포 차이를 설명할 수 있다.
2. CI와 CD를 입문 수준에서 구분할 수 있다.
3. workflow 파일이 어떤 역할을 하는지 설명할 수 있다.
4. build, test, deploy가 어떤 순서로 이어지는지 말할 수 있다.
5. 배포 후 확인 단계도 자동화 흐름의 일부라는 점을 이해할 수 있다.
6. 자동화가 왜 운영 비용과 실수를 줄이는지 설명할 수 있다.

## 3. 이번 시퀀스는 어디서부터 시작하는가

이번 시퀀스는 **09 시퀀스 answer 기준 레포를 바탕으로 확장**합니다.

즉, 아래는 이미 존재한다고 가정합니다.

- Spring Boot 프로젝트
- Docker 기반 실행 흐름
- 운영용 profile 또는 환경변수 분리 감각
- 서버에서 애플리케이션 실행 가능 상태

이번 시퀀스에서는 이 구조를 완전히 새로 만들지 않고,
**반복되던 빌드/테스트/배포 단계를 자동화 흐름으로 바꾸는 것**에 집중하세요.

## 5. 이번 시퀀스에서 TODO를 넣을 파일

학생이 직접 수정하는 핵심 파일은 아래로 제한하세요.

- `.github/workflows/deploy.yml`
- `deploy.sh` 또는 배포 스크립트 일부

필요하면 아래 같은 파일을 추가로 둘 수는 있지만,
핵심 TODO는 위 파일에 집중되어야 합니다.

- `.github/workflows/ci.yml`
- `scripts/check-deploy.sh`
- `.env.ci.example`

단, 범위를 과하게 넓히지 말고 “build → test → deploy → 확인” 흐름에만 집중하세요.

## 7. 이번 시퀀스의 중요한 제약

아래 제약을 반드시 지키세요.

- GitHub Actions의 모든 기능을 설명하지 마세요.
- matrix build, reusable workflows, self-hosted runner 같은 고급 주제를 넣지 마세요.
- Blue-Green, Canary 같은 배포 전략을 넣지 마세요.
- IaC, Kubernetes, Helm 같은 고급 운영 자동화는 넣지 마세요.
- 이번 시퀀스의 핵심은 **가장 단순한 CI/CD 흐름**입니다.
- build/test/deploy를 한 파일에서 보여줘도 되지만, 지나치게 복잡하게 만들지 마세요.
- secret 값은 실제 값이 아니라 예시 구조로만 보여주세요.
- 문서는 학생용 복습과 강사용 PPT 준비까지 같이 고려해서 쓰세요.
- 이번 응답에서는 오직 10 시퀀스 문서와 starter 코드 구조만 만드세요.

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```yaml
# TODO: CI/CD 설정
````

좋은 예:

```yaml
# TODO 1. 애플리케이션 빌드 step을 추가하세요.
# TODO 2. 테스트 실행 step을 연결하세요.
# TODO 3. 빌드와 테스트가 끝난 뒤 deploy step이 이어지게 만드세요.
# TODO 4. 마지막에는 배포 후 확인 단계도 넣어보세요.
```

배포 스크립트 예:

```bash
# TODO 1. 서버에서 최신 이미지를 받거나 실행 파일을 갱신하세요.
# TODO 2. 기존 프로세스 또는 컨테이너를 정리하세요.
# TODO 3. 새 버전을 실행하세요.
# TODO 4. 실행 후 정상 동작 여부를 확인하세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* 자동화의 핵심이 "반복 가능성"이라는 점이 드러나게 적는다.

예:

```text
# TODO: deploy만 연결하고 끝내지 마세요.
# 배포 후 확인 단계가 있어야 자동화 흐름이 더 완성됩니다.
```

또는

```text
# TODO: secret 값을 workflow 파일에 직접 쓰지 마세요.
# 민감한 값은 GitHub Secrets 같은 외부 설정으로 분리해야 합니다.
```

## 14. starter 코드 파일 작성 규칙

starter 파일은 아래 성격을 가져야 합니다.

* 바로 읽고 수정할 수 있어야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* 자동화 흐름을 한눈에 볼 수 있어야 함

필수 starter 파일 예시:

* `.github/workflows/deploy.yml`
* `deploy.sh` 또는 `scripts/deploy.sh`

필요하면 추가 가능:

* `.github/workflows/ci.yml`
* `scripts/check-deploy.sh`
* `.env.ci.example`

단, 범위를 과하게 넓히지 마세요.
