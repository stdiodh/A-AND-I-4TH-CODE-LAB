# 04. 인증과 JWT

## 목표

회원가입과 로그인 흐름을 만들고, JWT로 인증된 사용자를 구분합니다.
공개 API와 보호 API의 차이를 코드와 실행 결과로 확인합니다.

## 이 시퀀스에서 배우는 것

- 회원가입 요청 처리
- 비밀번호 암호화
- 로그인 성공 시 JWT 발급
- 인증 필터와 Security 설정
- 보호된 API에서 현재 사용자 확인
- 게시글 작성자 기준 인가 확인

## 테스트 방법

```bash
./gradlew test
```

테스트가 확인하는 것:

- 회원가입 또는 로그인 성공 시 JWT가 발급되는지 확인합니다.
- 로그인 실패가 성공 응답으로 처리되지 않는지 확인합니다.
- 인증 없이 보호 API에 접근하면 401로 실패하는지 확인합니다.
- 토큰을 포함하면 보호 API 접근이 성공하는지 확인합니다.
- 작성자가 아닌 사용자의 수정/삭제 요청은 403으로 실패하는지 확인합니다.

실패하면 먼저 볼 것:

- 테스트 요청의 `Authorization: Bearer <token>` 헤더가 올바른지 확인합니다.
- SecurityConfig에서 공개 API와 보호 API 경계가 의도와 같은지 확인합니다.
- 401은 인증 실패, 403은 인가 실패로 나누어 실패 메시지를 읽습니다.

완료 기준:

- 로그인 성공/실패 테스트가 통과합니다.
- 보호 API의 401 테스트와 토큰 접근 성공 테스트가 통과합니다.
- 작성자 인가 정책 테스트가 통과합니다.

## 확인할 API 또는 화면

- Swagger: `http://localhost:8080/swagger`
- `POST /auth/signup`
- `POST /auth/login`
- `GET /auth/me`
- `GET /posts`, `GET /posts/{id}`: 공개 API
- `POST /posts`: 인증 필요
- `PUT /posts/{id}`, `DELETE /posts/{id}`: 작성자만 가능
- Authorization 헤더가 있는 요청과 없는 요청

## 자주 발생하는 문제

- 비밀번호를 평문으로 저장하려고 합니다. 저장 전 암호화해야 합니다.
- JWT 발급과 인증을 같은 단계로 생각합니다. 발급은 로그인 결과이고, 인증은 다음 요청에서 일어납니다.
- 보호 API에 토큰 헤더를 넣지 않습니다. `Authorization: Bearer <token>` 형태를 확인합니다.
- 인증과 인가를 같은 의미로 사용합니다. 인증은 "누구인지" 확인하고, 인가는 "이 게시글을 바꿀 수 있는지" 확인합니다.
- 공개 API와 보호 API 경계를 명확히 보지 않습니다. 조회는 공개, 쓰기는 인증/인가 흐름으로 나눕니다.

## 완료 기준

- 회원가입과 로그인이 동작합니다.

- 인증 실패 응답을 확인했습니다.
