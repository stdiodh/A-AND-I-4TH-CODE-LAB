# 04. 인증과 JWT

## 시퀀스 목적

이 시퀀스는 `03`의 안전한 요청 처리 흐름 위에
회원가입, 로그인, JWT, 보호된 API 접근을 붙여
학생이 아래 흐름을 선명하게 이해하게 만드는 단계입니다.

- 회원가입 요청을 받는다
- 비밀번호를 안전하게 저장한다
- 로그인 요청을 처리한다
- 로그인 성공 후 JWT를 발급한다
- 보호된 API는 토큰이 있어야 접근할 수 있다
- 토큰에서 현재 사용자 정보를 읽을 수 있다

즉, 이번 시퀀스의 핵심은 Spring Security 문법을 많이 외우는 것이 아니라
인증은 누구인지 확인하는 것, 인가/보호는 접근 가능 여부를 나누는 것,
JWT는 로그인 이후 요청을 구분하는 수단 중 하나라는 감각을
코드와 실행 결과로 이해하는 것입니다.

## 학생이 완성해야 하는 최종 이해

학생은 이 시퀀스를 마친 뒤 아래를 설명할 수 있어야 합니다.

1. 인증과 인가의 차이
2. 회원가입과 로그인 최소 흐름
3. 비밀번호를 원문 그대로 저장하면 안 되는 이유
4. 로그인 후 JWT가 발급되는 흐름
5. 토큰 유무에 따라 보호된 API 결과가 달라지는 이유
6. 인증과 인가가 언제 갈리는지

## 학생 구현 순서

구현 순서는 반드시 아래 순서를 따릅니다.

1. `User` 핵심 필드를 확인합니다.
2. 회원가입 `Request DTO`를 만듭니다.
3. 회원가입 Service에서 비밀번호 인코딩을 연결합니다.
4. 로그인 `Request DTO`를 만듭니다.
5. 로그인 Service에서 사용자 조회와 비밀번호 확인을 연결합니다.
6. JWT 발급 메서드를 연결합니다.
7. 인증이 필요한 API를 지정합니다.
8. 토큰에서 사용자 정보를 읽는 흐름을 연결합니다.
9. 토큰 유무에 따른 결과 차이를 확인합니다.

문서와 TODO도 반드시 이 순서를 기준으로 설계합니다.

## 실무 확장 개념

이번 시퀀스의 실무 확장 개념은 아래입니다.

- 인증과 인가의 분리
- 역할 기반 접근

중요한 기준은 이렇습니다.

- 이번 시퀀스의 메인 구현 흐름은 회원가입 -> 로그인 -> JWT -> `/auth/me` 보호로 유지합니다.
- 대신 이론 문서에서는 로그인만 되면 모든 API를 열어두면 왜 부족한지를 보여줍니다.
- 그리고 `ROLE_ADMIN`, `본인만 접근 가능` 같은 인가 규칙이 어떤 코드 모양으로 이어지는지까지 설명합니다.

예를 들어 문서에는 아래 흐름이 들어가야 합니다.

1. 로그인은 성공했지만 아무 API나 다 열리면 안 되는 문제 상황
2. `/auth/me`가 왜 인증이고, `/admin/users` 같은 API는 왜 인가까지 필요할 수 있는지
3. `authenticated()`만으로는 부족한 이유
4. `hasRole("ADMIN")` 또는 `user.email == principal.name` 같은 인가 코드 예시
5. 이번 시퀀스에서 어디까지 구현하고, 어디까지는 설명만 하는지

즉 `04`의 실무 확장 개념은
JWT 발급 소개에서 끝나는 것이 아니라
인증과 인가가 언제 갈리고 어떤 규칙 코드로 확장되는지까지 같이 읽는 입문 단계로 설계합니다.

## 구현 제약

- OAuth2를 넣지 않습니다.
- Email Verification을 넣지 않습니다.
- refresh token까지 확장하지 않습니다.
- Role 체계를 메인 구현 범위로 복잡하게 확장하지 않습니다.
- 테스트 확장을 본격적으로 넣지 않습니다.
- Security 설정을 실무 풀버전처럼 길게 만들지 않습니다.
- 이번 시퀀스의 목표는 `회원가입 -> 로그인 -> JWT -> 보호된 API 접근`입니다.
- 실무 확장 개념인 인가/역할 기반 접근은 문서에서 상세히 다룹니다.
- 단, 문서에는 반드시 `문제 상황 -> 기본 인증의 한계 -> 인가 코드 예시`가 함께 들어가야 합니다.

## 수업 메시지

이번 단계에서 학생이 꼭 가져가야 할 문장은 아래입니다.

- 로그인 성공은 끝이 아니라, 다음 요청을 구분하기 위한 시작입니다.
- 비밀번호는 저장보다 안전한 저장이 더 중요합니다.
- JWT는 로그인 이후 요청에서 사용자를 구분하는 수단입니다.
- 인증은 “누구인지 확인”, 인가는 “이 기능을 써도 되는지 확인”입니다.
- 로그인만으로는 충분하지 않은 순간이 오고, 그때 역할 기반 접근이나 본인 확인 규칙이 필요해집니다.
