당신은 A&I 백엔드 커리큘럼용 실습 레포와 문서를 설계하는 Codex입니다.

지금부터는 전체 커리큘럼 중 **06. 테스트와 검증** 시퀀스만 다룹니다.
다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 06 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스들에서 만든 기능 위에,
학생이 아래 흐름을 직접 연결하면서 **테스트가 왜 필요한지**를 현실적으로 이해하게 만드는 것입니다.

- 이미 만든 Service 흐름을 테스트로 검증한다
- 정상 케이스와 예외 케이스를 나눠서 본다
- 검증 또는 인증 흐름도 테스트 대상이 될 수 있음을 본다
- 코드 변경 이후에도 기존 기능을 다시 신뢰할 수 있음을 경험한다

즉, 이번 시퀀스는 TDD 이론을 깊게 배우는 단계가 아닙니다.
학생이 **테스트는 기능 확인 도구이면서 변경 방지 장치**라는 감각을 갖게 만드는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. 테스트가 왜 필요한지 설명할 수 있다.
2. 정상 케이스와 실패 케이스를 구분해서 테스트할 수 있다.
3. Service 단위 테스트를 작성할 수 있다.
4. 검증 또는 인증 흐름도 테스트 대상으로 볼 수 있다.
5. 테스트가 깨졌을 때 원인을 찾고 수정한 뒤 다시 실행할 수 있다.
6. 변경 후에도 기존 기능을 다시 확인할 수 있다는 점을 이해할 수 있다.

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: 테스트 작성
````

좋은 예:

```kotlin
// TODO 1. given: 요청 DTO를 준비하세요.
// TODO 2. when: service.create(...)를 호출하세요.
// TODO 3. then: 제목과 내용이 기대값과 같은지 검증하세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 무엇을 먼저 봐야 하는지 드러나게 적는다.
* 정상 테스트와 예외 테스트의 차이를 분명히 느끼게 한다.

예:

```kotlin
// TODO: 예외가 발생하는 상황을 일부러 만들어보세요.
// 이 테스트의 목표는 실패를 막는 것이 아니라, 실패가 예상대로 일어나는지 확인하는 것입니다.
```

## 14. starter 코드 파일 작성 규칙

starter 코드는 아래 성격을 가져야 합니다.

* 바로 실행은 가능해야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* 테스트 실패/성공을 직접 확인할 수 있어야 함

필수 starter 파일 예시:

* `src/test/kotlin/.../service/PostServiceTest.kt`
* `src/test/kotlin/.../service/AuthServiceTest.kt`
* `src/test/kotlin/.../fixture/TestFixtureFactory.kt`

필요하면 추가 가능:

* `PostFixtureFactory.kt`
* `UserFixtureFactory.kt`

단, 범위를 과하게 넓히지 마세요.
