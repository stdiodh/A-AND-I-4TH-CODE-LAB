## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스에서 만든 CRUD 흐름 위에,
학생이 아래 흐름을 직접 연결하면서 **안전한 요청 처리와 실패 대응**을 이해하게 만드는 것입니다.

- 요청/응답 DTO를 분리한다
- Entity를 그대로 요청/응답 모델로 쓰지 않는다
- 잘못된 입력은 초반에 검증한다
- 실패도 일관된 형식으로 응답한다
- 비즈니스 예외와 검증 실패를 구분해서 본다

즉, 이번 시퀀스는 DTO, Validation, Exception Handling을 각각 따로 외우는 단계가 아닙니다.
학생이 **사용자 입력을 어디까지 믿을 수 있는지**, **왜 실패 응답도 설계 대상인지**를
코드와 실행 결과를 통해 이해하게 만드는 단계입니다.

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. 왜 Entity를 그대로 요청/응답에 쓰지 않는지 설명할 수 있다.
2. Request DTO와 Response DTO를 분리할 수 있다.
3. Validation이 어디서 동작하는지 설명할 수 있다.
4. 검증 실패와 비즈니스 예외를 구분해서 말할 수 있다.
5. 실패 응답을 왜 통일하는지 설명할 수 있다.
6. 정상 흐름과 실패 흐름을 직접 실행해서 확인할 수 있다.

## 4. 이번 시퀀스에서 학생이 직접 구현할 순서

구현 순서는 반드시 아래 순서를 기준으로 설계하세요.

1. `Request DTO`와 `Response DTO`를 분리한다.
2. Controller 입력 타입을 DTO로 바꾼다.
3. DTO → Entity 변환 코드를 작성한다.
4. `@Valid`를 붙인다.
5. 검증 실패 응답 구조를 만든다.
6. 비즈니스 예외 1개를 만든다.
7. 전역 예외 핸들러를 연결한다.
8. 실패 요청을 직접 실행해본다.

이 순서는 바꾸지 마세요.
문서와 코드 TODO도 반드시 이 순서에 맞게 설계하세요.

## 7. 이번 시퀀스의 중요한 제약

아래 제약을 반드시 지키세요.

- Security, JWT, 인증/인가를 넣지 마세요.
- 테스트를 본격적으로 넣지 마세요.
- ResponseEntity 공통 래퍼를 과도하게 복잡하게 만들지 마세요.
- 예외 구조를 실무 풀버전처럼 무겁게 만들지 마세요.
- 커스텀 validation을 과하게 확장하지 마세요.

- Entity를 요청/응답에 직접 쓰는 흐름을 DTO로 바꾸는 것이 중요합니다.
- Validation은 “요청 초입에서 잘못된 입력을 막는 장치”로 설명해야 합니다.
- Exception Handling은 “에러를 없애는 것”이 아니라 “일관된 응답 형식을 만드는 것”으로 설명해야 합니다.
- 문서는 학생용 복습과 강사용 PPT 준비까지 같이 고려해서 쓰세요.

## 9. 코드 설계 규칙

이번 시퀀스의 코드는 아래 원칙을 따라야 합니다.

- Kotlin + Spring Boot 기준
- 기존 CRUD 구조를 기반으로 확장
- Entity는 기존 것을 사용하되, 요청/응답에는 DTO를 사용
- Request DTO와 Response DTO는 분리
- Controller는 DTO를 입력으로 받도록 변경
- Service는 DTO → Entity 변환 또는 필요한 흐름을 담당
- Validation은 `@Valid`, `@NotBlank` 같은 가장 기본적인 수준으로만 사용
- Exception은 1~2개 수준으로 단순하게 유지
- ErrorResponse는 가장 기본적인 구조로 유지
- 전역 예외 핸들러는 최소한의 핵심 흐름만 보여줄 정도로 작성
- 코드 길이는 짧고 역할이 잘 보이게 유지

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: DTO 구현
````

좋은 예:

```kotlin
// TODO 1. 요청으로 받을 필드만 남기세요.
// TODO 2. id처럼 서버가 관리하는 값은 Request DTO에 넣지 마세요.
// TODO 3. title, content에 기본 검증 어노테이션을 붙여보세요.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* 역할이 섞이지 않게 유도한다.

예:

```kotlin
// TODO: Entity를 그대로 응답으로 반환하지 마세요.
// Response DTO로 바꿔서 필요한 값만 내려주세요.
```

또는

```kotlin
// TODO: ExceptionHandler에서 비즈니스 로직을 처리하지 마세요.
// 여기서는 실패 응답 형식을 정리하는 데 집중하세요.
```

## 14. starter 코드 파일 작성 규칙

starter 코드는 아래 성격을 가져야 합니다.

* 바로 실행은 가능해야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* Swagger에서 정상/실패 요청을 둘 다 확인할 수 있어야 함

필수 starter 파일 예시:

* `src/main/kotlin/.../dto/PostCreateRequest.kt`
* `src/main/kotlin/.../dto/PostUpdateRequest.kt`
* `src/main/kotlin/.../dto/PostResponse.kt`
* `src/main/kotlin/.../service/PostService.kt`
* `src/main/kotlin/.../exception/ErrorResponse.kt`
* `src/main/kotlin/.../exception/GlobalExceptionHandler.kt`

필요하면 추가 가능:

* `PostNotFoundException.kt`

기존 자산 재사용 가능:

* `PostEntity.kt`
* `PostRepository.kt`
* `PostController.kt`

단, Controller는 입력 타입을 DTO 기준으로 바꾸는 방향으로 조정하세요.
