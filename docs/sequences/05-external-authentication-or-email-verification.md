당신은 A&I 백엔드 커리큘럼용 실습 레포와 문서를 설계하는 Codex입니다.

지금부터는 전체 커리큘럼 중 **05. 외부 인증 또는 이메일 인증** 시퀀스만 다룹니다.
다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 05 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스에서 만든 회원가입/로그인/JWT 흐름 이후,
학생이 아래 두 확장 흐름 중 **하나를 끝까지 연결하면서**
인증이 실제 서비스에서 어떻게 확장되는지 이해하게 만드는 것입니다.

선택 가능한 메인 주제는 둘 중 하나입니다.

1. OAuth2 로그인
2. Email Verification

이번 시퀀스에서는 **둘 다 동시에 구현하지 마세요.**
하나를 메인으로 선택해 끝까지 연결하는 것이 핵심입니다.

즉, 이번 시퀀스는 인증 기술을 많이 보는 단계가 아니라,
학생이 **기존 로그인 흐름 바깥에서 외부 인증 또는 계정 검증 흐름이 어떻게 붙는지**
입문 수준에서 경험하게 만드는 단계입니다.

---

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래 중 하나를 할 수 있어야 합니다.

### OAuth2를 메인으로 선택한 경우
1. 자체 로그인과 외부 로그인 차이를 설명할 수 있다.
2. 외부 로그인 성공 후 사용자 정보를 읽는 흐름을 설명할 수 있다.
3. 신규 사용자와 기존 사용자를 어떻게 구분하는지 말할 수 있다.
4. 외부 인증 결과를 우리 서비스 사용자와 연결할 수 있다.

### Email Verification을 메인으로 선택한 경우
1. 이메일 인증이 왜 필요한지 설명할 수 있다.
2. 인증 토큰을 발급하고 검증하는 흐름을 설명할 수 있다.
3. 인증 링크를 클릭했을 때 어떤 일이 일어나는지 말할 수 있다.
4. 계정 신뢰를 높이기 위한 검증 흐름을 이해할 수 있다.

## 3. 이번 시퀀스는 어디서부터 시작하는가

이번 시퀀스는 **04 시퀀스 answer 기준 레포를 바탕으로 확장**합니다.

즉, 아래는 이미 존재한다고 가정합니다.

- Spring Boot 프로젝트
- Swagger 설정
- DB 연결
- User 기반 회원가입/로그인 흐름
- PasswordEncoder
- JWT 발급
- 보호된 API 최소 흐름

이번 시퀀스에서는 이 구조를 완전히 새로 만들지 않고,
**기존 인증 흐름을 외부 인증 또는 이메일 인증으로 확장하는 것**에 집중하세요.

## 8. 이번 시퀀스의 중요한 제약

아래 제약을 반드시 지키세요.

- OAuth2와 Email Verification을 동시에 구현하지 마세요.
- JWT를 다시 깊게 확장하지 마세요.
- refresh token, social account linking 고급 주제까지 확장하지 마세요.
- 이메일 인증 만료 정책, 재전송 전략, 복잡한 보안 정책까지 깊게 가지 마세요.

- 외부 인증 제공자별 차이를 길게 설명하지 마세요.
- 메인 주제 하나를 선택하고 그 흐름만 끝까지 연결하세요.
- 이번 시퀀스의 핵심은 “확장 인증 흐름 하나를 완성하는 것”입니다.
- 문서는 학생용 복습과 강사용 PPT 준비까지 같이 고려해서 쓰세요.
- 이번 응답에서는 오직 05 시퀀스 문서와 starter 코드 구조만 만드세요.

## 11. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: OAuth2 구현
````

좋은 예:

```kotlin id="1k5u99"
// TODO 1. 외부 로그인 성공 후 받은 사용자 정보를 읽으세요.
// TODO 2. email 기준으로 기존 사용자인지 확인하세요.
// TODO 3. 없으면 새 사용자를 생성하고, 있으면 기존 사용자를 사용하세요.
// TODO 4. 우리 서비스 기준 응답으로 정리하세요.
```

Email Verification 예:

```kotlin id="65g5lv"
// TODO 1. 인증 토큰을 생성하세요.
// TODO 2. 이 토큰을 포함한 인증 링크를 만드세요.
// TODO 3. 메일 발송 Service를 호출하세요.
// TODO 4. 아직 실제 메일 인프라 전체를 구현하는 것이 목표는 아닙니다.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* 역할이 섞이지 않게 유도한다.

예:

```kotlin id="my9ps4"
// TODO: Controller에서 인증 전체 로직을 직접 처리하지 마세요.
// Service가 핵심 흐름을 맡도록 유지하세요.
```

## 15. starter 코드 파일 작성 규칙

starter 코드는 아래 성격을 가져야 합니다.

* 바로 실행은 가능해야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* 선택한 주제의 성공 흐름을 확인할 수 있어야 함

### OAuth2 선택 시 starter 파일 예시

* `src/main/kotlin/.../security/OAuthUserService.kt`
* `src/main/kotlin/.../controller/AccountController.kt`
* `src/main/kotlin/.../service/UserService.kt`

### Email Verification 선택 시 starter 파일 예시

* `src/main/kotlin/.../service/MailVerificationService.kt`
* `src/main/kotlin/.../domain/VerificationToken.kt`
* `src/main/kotlin/.../controller/AccountController.kt`

필요하면 추가 가능하지만, 범위를 과하게 넓히지 마세요.

## 16. checklist.md 작성 규칙

docs/checklist.md 에는 반드시 아래를 넣으세요.

### 학생 체크리스트

예:

* 자체 로그인과 외부 인증 차이를 설명할 수 있다.
* 또는 이메일 인증이 왜 필요한지 설명할 수 있다.
* 인증 흐름이 실제 서비스에 어떻게 연결되는지 설명할 수 있다.

### 강사 / PPT 체크리스트

예:

* 이 시퀀스에서 두 개를 다 하지 않는다는 기준이 명확한가
* OAuth2 또는 Email Verification 중 무엇을 메인으로 할지 정했는가
* 학생이 따라칠 범위를 과하게 잡지 않았는가

또 아래도 포함하세요.

* 수업 전 준비 체크
* 수업 중 질문 체크
* 수업 후 복습 체크

## 17. assets.md 작성 규칙
