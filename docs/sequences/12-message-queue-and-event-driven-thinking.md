당신은 A&I 백엔드 커리큘럼용 실습 레포와 문서를 설계하는 Codex입니다.

지금부터는 전체 커리큘럼 중 **12. 이벤트 기반 사고 확장** 시퀀스만 다룹니다.
다른 시퀀스 내용은 절대 섞지 말고, 이번 요청에서는 12 시퀀스에 필요한 레포 구조, 학생용 starter 코드, 정답 가이드, 이론 문서만 정확하게 만드세요.

## 1. 이번 작업의 목표

이번 시퀀스의 목적은 이전 시퀀스들에서 만든 요청/응답 중심 구조 위에,
학생이 아래 흐름을 직접 연결하면서 **이벤트 기반 흐름이 왜 필요한지**를 이해하게 만드는 것입니다.

- 어떤 동작이 발생한다
- 그 결과를 이벤트로 발행한다
- 다른 쪽에서 그 이벤트를 소비한다
- 동기 호출과 비동기 전달의 차이를 비교한다

즉, 이번 시퀀스는 Kafka나 RabbitMQ 전체를 깊게 배우는 단계가 아닙니다.
학생이 **시스템이 커질 때 왜 직접 호출만으로는 불편해질 수 있는지**,
그리고 **이벤트 기반 흐름이 어떤 문제를 완화할 수 있는지**를 입문 수준에서 이해하게 만드는 단계입니다.

이 시퀀스의 목표는 “완전 구현”이 아니라 **시야 확장**입니다.

---

## 2. 이번 시퀀스에서 학생이 완성해야 하는 최종 상태

학생은 이 시퀀스를 마친 뒤 아래를 할 수 있어야 합니다.

1. 동기 호출과 비동기 전달 차이를 설명할 수 있다.
2. 이벤트가 무엇인지 입문 수준에서 설명할 수 있다.
3. 이벤트 발행과 이벤트 소비 흐름을 말할 수 있다.
4. 메시지 큐가 왜 필요한지 말할 수 있다.
5. 이벤트 기반 구조가 어떤 문제를 완화할 수 있는지 이해할 수 있다.
6. MSA는 무조건 쓰는 것이 아니라는 점을 말할 수 있다.

## 3. 이번 시퀀스는 어디서부터 시작하는가

이번 시퀀스는 **11 시퀀스 answer 기준 레포를 바탕으로 확장**합니다.

즉, 아래는 이미 존재한다고 가정합니다.

- Spring Boot 프로젝트
- CRUD / DTO / Validation / Exception Handling
- 인증 / 테스트 / 캐시 / 실시간 통신 / 배포 / 자동화 / 리팩토링까지의 기본 감각
- 하나의 애플리케이션 안에서 기능이 연결되는 구조

이번 시퀀스에서는 이 구조를 완전히 새로 만들지 않고,
**하나의 기능 결과를 이벤트로 분리해서 다른 흐름으로 넘겨보는 것**에 집중하세요.

## 7. 이번 시퀀스의 중요한 제약

아래 제약을 반드시 지키세요.

- Kafka, RabbitMQ, SQS 등 특정 기술의 고급 운영 지식을 깊게 설명하지 마세요.
- 파티션, consumer group, offset 관리 같은 고급 개념을 깊게 넣지 마세요.
- MSA 전체 설계를 구현하려고 하지 마세요.
- Saga, Outbox, CDC 같은 고급 패턴을 넣지 마세요.
- 이번 시퀀스의 핵심은 **가장 단순한 이벤트 발행/소비 흐름**입니다.
- “주문 생성 → 알림 발행”처럼 단순한 예시 1개에 집중하세요.
- 메시지 브로커 운영 전체가 아니라 흐름 이해가 목적이라는 점을 분명히 하세요.
- 문서는 학생용 복습과 강사용 PPT 준비까지 같이 고려해서 쓰세요.
- 이번 응답에서는 오직 12 시퀀스 문서와 starter 코드 구조만 만드세요.

## 9. 코드 설계 규칙

이번 시퀀스의 코드는 아래 원칙을 따라야 합니다.

- Kotlin + Spring Boot 기준
- 이벤트 발행/소비 흐름이 한눈에 보이게 유지
- 예시는 “주문 생성 → 알림 전송” 또는 그에 준하는 단순 사례 1개만 사용
- 이벤트 DTO는 최소 필드만 사용
  - 예: orderId, userId, message
- 발행자는 이벤트를 만들고 전달하는 역할
- 소비자는 이벤트를 받아 후속 동작을 수행하는 역할
- 코드 길이는 짧고 역할이 잘 보이게 유지
- 실제 메시지 브로커가 있더라도 고급 설정은 숨기고 흐름 중심으로 보여주세요
- 필요하면 “브로커 연결은 제공, 학생은 발행/소비 흐름만 구현” 형태로 설계하세요

## 10. TODO 작성 규칙

TODO는 반드시 순서형 힌트로 작성하세요.

좋지 않은 예:
```kotlin
// TODO: 이벤트 처리
````

좋은 예:

```kotlin id="5a8v1i"
// TODO 1. 주문이 생성되면 필요한 값을 담은 이벤트 객체를 만드세요.
// TODO 2. 이 이벤트를 발행 서비스로 전달하세요.
// TODO 3. 소비자 쪽에서 이 이벤트를 받아 알림 흐름으로 연결하세요.
// TODO 4. 지금은 이벤트 전달 흐름을 이해하는 것이 목표입니다.
```

또 아래 원칙을 반드시 지키세요.

* `TODO 1`, `TODO 2`, `TODO 3`처럼 번호를 붙인다.
* 한 TODO에는 한 동작만 넣는다.
* 문법 설명보다 흐름 설명을 적는다.
* 학생이 하지 말아야 할 것도 적는다.
* 동기 호출과 비동기 흐름 차이가 드러나게 적는다.

예:

```kotlin id="mujijj"
// TODO: 이벤트 소비자에서 주문 생성 로직까지 다시 처리하지 마세요.
// 이번 단계의 핵심은 '발행된 결과를 받아 후속 작업을 분리하는 흐름'입니다.
```

또는

```kotlin id="0ohrj3"
// TODO: MSA 전체 구현처럼 확장하지 마세요.
// 지금은 하나의 이벤트가 다른 흐름으로 전달되는 최소 구조만 보면 충분합니다.
```

## 14. starter 코드 파일 작성 규칙

starter 파일은 아래 성격을 가져야 합니다.

* 바로 읽고 수정할 수 있어야 함
* 핵심 부분은 비어 있거나 TODO로 남아 있어야 함
* TODO는 학생이 수업 시간 안에 채울 수 있어야 함
* TODO 없는 보일러플레이트는 제공
* 이벤트 흐름을 직접 확인할 수 있어야 함

필수 starter 파일 예시:

* `src/main/kotlin/.../event/OrderCreatedEvent.kt`
* `src/main/kotlin/.../service/EventPublisherService.kt`
* `src/main/kotlin/.../consumer/NotificationConsumer.kt`

필요하면 추가 가능:

* `OrderService.kt`
* `NotificationService.kt`
* `EventConfig.kt`

단, 범위를 과하게 넓히지 마세요.
