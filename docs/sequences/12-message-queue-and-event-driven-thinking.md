# 12. 이벤트 기반 사고 확장

## 시퀀스 목적

이 시퀀스는 지금까지의 요청/응답 중심 구조 위에
`결과를 이벤트로 분리해 다른 흐름으로 넘기는 방식`을 처음 소개하는 단계입니다.

핵심은 고급 메시징 운영을 배우는 것이 아니라,
학생이 아래 질문에 답할 수 있게 만드는 것입니다.

- 왜 직접 호출만으로는 점점 불편해질 수 있을까
- 이벤트는 어떤 역할을 할까
- 메시지 큐는 발행자와 소비자를 어떻게 덜 묶어줄까
- 서비스가 분리된 구조라면 이벤트가 큐를 통해 어떻게 이동할까

즉, 이번 시퀀스의 목표는 “완전한 MSA 구현”이 아니라
이벤트 기반 사고를 입문 수준에서 한 번 확장해보는 것입니다.

## 학생이 완성해야 하는 최종 이해

학생은 이 시퀀스를 마친 뒤 아래를 설명할 수 있어야 합니다.

1. 동기 호출과 비동기 전달의 차이
2. 이벤트가 무엇인지
3. 이벤트 발행과 소비 흐름
4. 메시지 큐가 왜 필요한지
5. 이벤트 기반 구조가 어떤 문제를 완화할 수 있는지
6. MSA가 무조건 답이 아니라는 점

## 시작 기준

이번 시퀀스는 `11-answer` 기준의 구조화된 Spring Boot 프로젝트 위에서 시작합니다.

이미 학생은 아래 감각을 가진 상태라고 가정합니다.

- CRUD / DTO / Validation / Exception Handling
- 인증 / 테스트 / 캐시 / 실시간 통신 / 배포 / 자동화
- 리팩토링을 통해 역할을 나누어 읽는 감각

이번 단계에서는 그 위에
`주문 생성 -> 이벤트 발행 -> 알림 소비`
예시 하나만 얹어서 흐름을 비교합니다.

여기서 MSA는 깊게 구현하지 않고,
“주문 서비스와 알림 서비스가 따로 있고, 그 사이를 이벤트 큐가 이어준다”는 관점까지만 소개합니다.

## 레포 전략

- 사용 레포: `spring-boot-event-driven-lab`
- 안내 브랜치: `main`
- 학생 starter: `12-implementation`
- 정답 브랜치: `12-answer`

이번 시퀀스는 이전 레포를 계속 확장하지 않고,
이벤트와 메시지 큐라는 새 학습 주제를 분리한 별도 레포로 운영합니다.

## 학생 구현 순서

구현 순서는 반드시 아래 순서를 따릅니다.

1. 이벤트 DTO를 만듭니다.
2. 이벤트 발행 코드를 만듭니다.
3. 이벤트 소비 코드를 만듭니다.
4. 흐름을 실행해봅니다.
5. 동기/비동기 차이를 비교합니다.

문서와 TODO도 이 순서를 기준으로 설계합니다.

## TODO를 넣는 핵심 파일

- `OrderCreatedEvent.kt`
- `EventPublisherService.kt`
- `NotificationConsumer.kt`

보조 파일은 미리 제공할 수 있지만,
핵심 TODO는 위 세 파일에 집중합니다.

## 미리 제공하는 것

- RabbitMQ 실행 환경
- Producer / Consumer 기본 틀
- `OrderService`, `NotificationService`, `EventConfig`
- build 설정
- application 설정
- 주문 생성 API와 알림 조회 API
- 실행 가능한 starter 환경

학생은 보일러플레이트보다
발행/소비 흐름을 손으로 따라치는 데 집중해야 합니다.

## 구현 제약

- Kafka, RabbitMQ 고급 운영 지식을 깊게 넣지 않습니다.
- partition, consumer group, offset 심화를 넣지 않습니다.
- Saga, Outbox, CDC 같은 고급 패턴을 넣지 않습니다.
- MSA 전체 설계를 구현하지 않습니다.
- 예시는 `주문 생성 -> 알림 소비` 하나로 제한합니다.
- 메시지 브로커 운영 전체가 아니라 흐름 이해가 목적임을 분명히 합니다.

## 검증 기준

- `12-implementation`은 TODO가 있어도 실행 가능한 starter여야 합니다.
- `12-answer`는 완성 코드와 정답 문서가 일치해야 합니다.
- starter와 answer 둘 다 `./gradlew test`가 통과해야 합니다.
- 루트 허브에는 서브모듈과 상태표가 함께 반영되어야 합니다.

## 수업 메시지

이번 단계에서 학생이 꼭 가져가야 할 문장은 아래입니다.

- 이벤트는 “어떤 일이 일어났다”는 사실을 다른 흐름에 전달하는 방법입니다.
- 메시지 큐는 발행자와 소비자를 느슨하게 연결해줍니다.
- 동기 요청/응답만이 유일한 연결 방식은 아닙니다.
- 서비스가 분리되면 이벤트가 큐를 통해 이동하는 구조를 자주 보게 됩니다.
- 그렇다고 이벤트 기반 구조가 언제나 정답인 것도 아닙니다.
