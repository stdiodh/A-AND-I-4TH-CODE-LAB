# 12. 메시지 큐와 이벤트 기반 사고

## 목표

요청/응답 중심 구조 위에 이벤트 발행과 처리 흐름을 붙입니다.
고급 메시징 운영이 아니라 이벤트가 발행자와 소비자를 덜 묶어주는 구조를 이해하는 것이 목표입니다.

## 이 시퀀스에서 배우는 것

- 동기 호출과 이벤트 전달의 차이
- 이벤트 DTO가 담아야 하는 최소 정보
- 메시지 큐의 역할
- 이벤트 발행자와 소비자 분리
- 이벤트 처리와 트랜잭션 경계를 함께 생각해야 하는 이유

## 시작 브랜치

```bash
git checkout 12-implementation
```

## 실습 전 확인

- 토픽 레포: `spring-boot-event-driven-lab`
- 가이드 브랜치: `main`
- 시작 브랜치: `12-implementation`
- 이전 기준: `11-answer`
- RabbitMQ 또는 메시지 브로커가 Docker로 실행될 수 있어야 합니다.
- 이번 시퀀스는 실제 MSA 전체 구현이 아닙니다.

## 실행 방법

```bash
docker compose up -d
./gradlew bootRun
```

## 테스트 방법

```bash
./gradlew test
```

테스트가 확인하는 것:

- 주문 생성 흐름에서 이벤트가 발행되는지 확인합니다.
- 이벤트 핸들러 또는 consumer가 호출되는지 확인합니다.
- 소비자가 이벤트를 알림 기록으로 변환하는지 확인합니다.
- 발행 실패와 영속 주문 저장의 트랜잭션 경계는 현재 구현 밖의 확장 과제로 구분합니다.

실패하면 먼저 볼 것:

- 메시지 브로커가 필요한 테스트인지, test double로 대체 가능한 테스트인지 구분합니다.
- 이벤트 payload에 후속 처리에 필요한 최소 필드가 들어 있는지 확인합니다.
- 현재 `AtomicLong` 기반 예제에는 영속 주문 저장이 없으므로 저장 성공을 테스트했다고 해석하지 않습니다.

완료 기준:

- 이벤트 발행 테스트가 통과합니다.
- 이벤트 핸들러 호출 테스트가 통과합니다.
- 현재 단위 테스트가 발행 호출과 소비 결과를 확인하는 범위를 설명할 수 있습니다.

## 확인할 API 또는 화면

- `POST /event-orders`
- `GET /event-orders/notifications`
- RabbitMQ 컨테이너 상태
- 이벤트 발행 로그
- 이벤트 소비 로그

## 자주 발생하는 문제

- 이벤트에 너무 많은 데이터를 담습니다. 이번 단계에서는 후속 작업에 필요한 최소 정보만 담습니다.
- 주문 서비스가 알림 처리까지 직접 맡습니다. 주문 서비스는 이벤트 발행까지만 담당합니다.
- 메모리 id 생성 예제를 영속 주문 저장으로 오해합니다. 저장과 발행의 원자성이 필요하면 outbox 같은 후속 패턴을 별도로 검토합니다.
- 메시지 큐를 무조건 좋은 구조로 이해합니다. 분리의 이점과 운영 복잡도를 함께 봅니다.

## 완료 기준

- 주문 생성 후 이벤트가 발행됩니다.
- 소비자가 이벤트를 받아 알림 흐름으로 연결합니다.
- 동기 호출과 이벤트 전달의 차이를 설명할 수 있습니다.
- 현재 예제에 영속 트랜잭션이 없다는 점과 실제 저장·발행 경계에서 생길 수 있는 문제를 말할 수 있습니다.
- `./gradlew test`가 통과합니다.
